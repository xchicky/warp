# M4.2: Local Agent Plan Mode

## Summary

Add a feature-flagged plan mode for the local OpenAI-compatible agent. In plan mode, the agent may inspect context with read-only tools and produce a plan, but it must not write files, execute shell commands, call mutating MCP tools, or otherwise change workspace state. Exiting plan mode requires an explicit `exit_plan_mode` tool call with the complete plan.

Current branch note: Warp already has `UserQueryMode::Plan`, `/plan` parsing, and query display support. The local direct agent currently does not enforce a read-only mode or expose an `exit_plan_mode` tool.

## User Story

As a Warp user, I want to ask the local agent to investigate and plan before making changes, so I can review the proposed approach before approving any file edits, shell commands, or external tool calls.

## Acceptance Criteria

1. Add `FeatureFlag::LocalAgentPlanMode`, default off, following the existing local-agent feature flag pattern.
2. When `LocalAgentPlanMode` is off, `/plan` and normal local OpenAI-compatible behavior remain unchanged from the post-M4.1 baseline.
3. When the flag is on, users can enter plan mode through existing explicit query mode where practical:
   - `/plan <query>` should map to `UserQueryMode::Plan`.
   - A UI entry point may be added only if it can reuse existing query-mode plumbing.
4. Agent-proposed entry into plan mode is allowed only after visible user approval. A provider suggestion alone must not silently switch modes.
5. Plan mode persists for the local conversation until `exit_plan_mode` completes and the user explicitly approves leaving plan mode or submits a normal follow-up that confirms execution. The local conversation id must not change during the mode transition.
6. Add a local OpenAI-compatible function tool named `exit_plan_mode` that is advertised only while plan mode is active.
7. `exit_plan_mode` requires:
   - `plan`: non-empty bounded markdown/plain text containing the complete proposed plan.
   - Optional `summary`: short bounded label for UI/history.
8. When `exit_plan_mode` succeeds:
   - Surface the plan to the user as assistant-visible output.
   - Mark the conversation as having a pending plan ready for approval or normal-mode continuation.
   - Stop the current provider loop without executing mutating tools in the same round.
9. Plan mode tool policy must be enforced in both tool advertisement and execution:
   - Hidden mutating tools must not be advertised.
   - Direct calls to forbidden tool names must fail closed even if the provider emits them.
10. Allowed tools in plan mode:
   - `Grep`
   - `ReadFiles`
   - `FileGlobV2`
   - `list_directory`
   - `suggest_shell_command`
   - `exit_plan_mode`
   - Todo read/list operation if M4.1 exposes one; otherwise current todo state may be included as read-only context.
11. Forbidden tools in plan mode:
   - `apply_file_diff`
   - `write_file`
   - `edit_file`
   - `run_shell_command`
   - Mutating `todo_write` updates.
   - MCP `CallMcpTool` by default, unless a future trusted read-only classification exists.
12. `suggest_shell_command` remains allowed because it proposes a command without executing it. Its copy should make clear that the command is suggested for later user approval, not run during plan mode.
13. Plan mode must use a system/executor-level policy object or equivalent hard check. Do not rely only on system prompt wording to prevent mutations.
14. If a provider attempts a forbidden tool call, return a bounded provider-visible error explaining that the tool is unavailable in plan mode and do not execute the tool.
15. Mixed tool batches in plan mode follow the existing ordered scheduler. If any forbidden call appears, fail that call closed without allowing later mutating work to run.
16. Tool-result resume/finalize rounds must remain routed to the local provider and preserve the same local conversation id.
17. History conversion must preserve enough state for plan-mode resume and `exit_plan_mode` finalization. It must not fall back to remote/Oz routes.
18. If the provider returns normal assistant prose without calling `exit_plan_mode`, keep the conversation in plan mode and prompt the provider/user path to finish with `exit_plan_mode` on the next round.
19. Existing local-agent tests continue to pass. Add focused tests for:
   - `/plan` enters local plan mode when the flag is on.
   - Tool list includes only allowed tools plus `exit_plan_mode`.
   - Forbidden tool calls are denied at execution even if emitted directly.
   - `exit_plan_mode` validates bounded plan text and exits without changing conversation id.
   - Tool-result resume stays local.
   - Plan mode does not run MCP tools by default.
   - Plan mode does not allow mutating TodoWrite operations.

## Safety Risks

- Prompt-injection bypass: provider output could try to call hidden tools. Mitigate with executor-level allow/deny checks.
- Accidental mutation: MCP tools may not declare read/write behavior reliably. Mitigate by denying MCP calls in M4.2 plan mode unless a trusted read-only classification is added later.
- Context loss: mode transitions could start a new conversation and lose prior reasoning/tool context. Mitigate by preserving local conversation id and route state.
- Confusing exit semantics: users may think a plan was approved automatically. Mitigate with explicit `exit_plan_mode` output and no automatic execution.
- Shell confusion: suggested commands could look executed. Mitigate with clear suggested-only UI/result copy.

## Scale Estimate

L. The feature is conceptually simple but crosses query mode parsing, local tool construction, scheduling, out-of-band tool denial, resume routing, and user-visible mode state.

## PR Split Suggestion

Keep M4.2 as one implementation PR only if the diff remains contained; otherwise split UI affordances into a follow-up. Target at most seven commits:

1. Add `LocalAgentPlanMode` feature flag, default off.
2. Add local plan-mode state derived from `UserQueryMode::Plan` and preserved across local resume.
3. Add plan-mode tool policy and filter advertised local/MCP tools.
4. Add executor-level forbidden-tool denial for direct provider calls.
5. Add `exit_plan_mode` schema, validation, output, and state transition.
6. Add history/resume conversion tests for local route and conversation id preservation.
7. Add UX copy or minimal UI wiring for plan-ready output if existing rendering is insufficient.

Each commit should compile independently. Keep implementation out of remote/Oz paths except for shared query-mode or rendering primitives.

## Entry Points

- `app/src/ai/agent/mod.rs`: `UserQueryMode`, `extract_user_query_mode`, and display helpers.
- `app/src/ai/agent/local/mod.rs`: local tool definition filtering, `execute_local_tool_batch`, forbidden tool denial, resume/finalize handling, and local tests.
- `app/src/ai/agent/api/convert_to.rs`, `convert_from.rs`, and `convert_conversation.rs`: query mode and history conversion.
- `app/src/ai/blocklist/controller.rs`: user query mode propagation into agent requests.
- `app/src/ai/blocklist/block/view_impl/common.rs`: existing `/plan` prefix highlighting and display behavior.
- `app/src/ai/blocklist/prompt/plan_and_todo_list.rs`: existing plan/todo UI surface if plan-ready output needs a visible affordance.
- `app/src/ai/mcp/*`: MCP passthrough policy integration if advertised tool filtering needs MCP context.
- `app/Cargo.toml`, `app/src/lib.rs`, `crates/warp_features/src/lib.rs`: feature flag plumbing.
