# M4.1: Local Agent TodoWrite Tool

## Summary

Add a feature-flagged `todo_write` tool so the local OpenAI-compatible agent can maintain a visible, conversation-scoped task list. Reuse Warp's existing todo model, history message, and UI rendering instead of adding a parallel local-only todo card.

Current branch note: Warp already has `AIAgentTodoList`, `TodoOperation`, `api::Message::UpdateTodos`, `BlocklistAIHistoryEvent::UpdatedTodoList`, todo rendering in AI blocks, and `AgentTodosPopupView`. The local direct agent currently does not expose a todo-writing tool.

## User Story

As a Warp user using a local OpenAI-compatible agent, I want the agent to maintain a clear task list while working through multi-step requests, so I can see what it plans to do, what is in progress, and what has been completed without relying only on prose updates.

## Acceptance Criteria

1. Add `FeatureFlag::LocalAgentTodoWrite`, default off, following the existing local-agent feature flag pattern.
2. When `LocalAgentTodoWrite` is off, local OpenAI-compatible tool definitions and request behavior are unchanged.
3. When the flag is on, advertise one OpenAI-compatible function tool named `todo_write`.
4. `todo_write` accepts a full replacement list rather than piecemeal mutation:
   - `todos`: required array of todo objects.
   - Each todo has `id`, `content`, and `status`.
   - `status` is one of `pending`, `in_progress`, or `completed`.
   - Optional bounded `summary` may explain the update.
5. Use a single full-list replacement contract so retries and provider replays are idempotent. Do not require the provider to know the previous local todo state to compute create/update/delete patches.
6. Validate inputs before updating conversation state:
   - Reject duplicate ids.
   - Reject empty or whitespace-only `id` or `content`.
   - Reject unknown statuses.
   - Reject more than one `in_progress` item.
   - Enforce bounded todo count and bounded string length.
7. Map the single `in_progress` todo to the first pending item in `AIAgentTodoList`, because existing Warp todo state derives in-progress from `pending_items.first()`.
8. Preserve provider order:
   - `in_progress` item first in pending items if present.
   - Remaining `pending` items follow in provider order.
   - `completed` items follow in provider order in completed state.
9. Emit existing todo history/state primitives:
   - Prefer `api::Message::UpdateTodos` and existing `TodoOperation` conversion paths.
   - Emit `BlocklistAIHistoryEvent::UpdatedTodoList` so existing UI updates.
   - Do not add a second local-only todo persistence mechanism.
10. Todo state is scoped to the current conversation. It must not be written to a global file, project file, user settings, or provider-specific cache. Existing conversation history replay may reconstruct the list within the same conversation.
11. Treat `todo_write` as a sequential local tool. It must not run in parallel with other sequential or mutating tools, including file writes, shell execution, and MCP passthrough calls.
12. If a provider batch includes multiple `todo_write` calls, apply them in provider order. The final list after the batch must match the last valid call that executed.
13. Tool results returned to the provider are bounded summaries:
   - Include counts by status and the current in-progress item title if present.
   - Do not echo arbitrarily large todo JSON back into provider context.
14. Invalid `todo_write` arguments produce a provider-visible local tool error and do not partially update conversation state.
15. Existing local tool schemas must not be shadowed. `todo_write` must not conflict with MCP namespaced tools such as `mcp__<server>__<tool>`.
16. UI should reuse existing todo rendering and popup where practical. If a new card is necessary, it must consume the same `AIAgentTodoList` state and not fork todo semantics.
17. Add focused tests for:
   - Feature flag off omits `todo_write`.
   - Feature flag on advertises the expected schema.
   - Valid full-list replacement updates pending, in-progress, and completed state.
   - Duplicate ids, empty content, unknown status, and multiple in-progress items fail closed.
   - Sequential scheduling preserves provider order with mixed mutating tools.
   - Conversation/history conversion replays todo updates.
   - Tool result summaries are bounded.

## Safety Risks

- Hidden persistence: todos could accidentally become cross-conversation memory. Mitigate by using conversation history/state only.
- State corruption: partial application on invalid input could desynchronize UI and provider context. Mitigate with validate-before-apply and atomic replacement.
- Parallel race: todo updates mixed with writes or MCP calls could appear out of order. Mitigate by marking `todo_write` sequential.
- Context blowup: large todo lists could flood provider history. Mitigate with count/string limits and bounded tool results.
- UI divergence: a local-only todo card could disagree with existing todo views. Mitigate by reusing existing todo primitives.

## Scale Estimate

M. Existing todo data structures and UI exist, but local direct needs a new tool schema, validation, history emission, sequential scheduling, and tests.

## PR Split Suggestion

Keep M4.1 as one implementation PR with at most six commits:

1. Add `LocalAgentTodoWrite` feature flag, default off.
2. Add `todo_write` schema and argument parser/validator.
3. Add conversion from local todo tool args to existing `AIAgentTodoList` and `api::Message::UpdateTodos`.
4. Integrate `todo_write` with local tool execution and sequential scheduling.
5. Reuse/update existing todo UI/history plumbing only where needed.
6. Add unit tests for schema, validation, state replay, bounded output, and ordered scheduling.

Each commit should compile independently. Avoid UI rewrites unless existing todo rendering cannot display local todo updates.

## Entry Points

- `app/src/ai/agent/local/mod.rs`: local tool definitions, `execute_local_tool_batch`, sequential tool classification, argument parsing, and local tests.
- `app/src/ai/agent/todos/mod.rs`: existing `AIAgentTodoList` state semantics.
- `app/src/ai/agent/mod.rs`: `AIAgentTodo`, `AIAgentTodoId`, and `TodoOperation`.
- `app/src/ai/agent/conversation.rs`: `update_todo_list_from_todo_op`, todo state replay, and `UpdatedTodoList` event emission.
- `app/src/ai/agent/api/convert_from.rs` and `app/src/ai/agent/api/convert_conversation.rs`: existing `Message::UpdateTodos` conversion and replay paths.
- `app/src/ai/blocklist/block/view_impl/todos.rs` and `app/src/ai/agent/todos/popup.rs`: existing todo rendering.
- `app/Cargo.toml`, `app/src/lib.rs`, `crates/warp_features/src/lib.rs`: feature flag plumbing.
