# M6.2 Option A Research Notes

## Scope

Research for M6.2 Option A: expose the configured Local OpenAI-compatible Agent from Full Terminal Use without building a synthetic CLI-agent byte-stream bridge.

PM decisions already fixed the product direction:

- Add a Local Agent option to the Full Terminal Use model selector.
- Reuse the existing Base Agent panel/chat route.
- Default context: cwd, selected text, foreground process name.
- Support `run_shell_command` through Warp's existing command action path, not foreground TUI stdin.
- Decide how conversation history works for FTU-local conversations that do not have a `CLIAgentSession`.

## Q1: FTU Selector Can Host Local Agent

The selector already has separate Base and Full Terminal Use tabs:

- `app/src/terminal/input/models/view.rs:59` defines `InlineModelSelectorTab::{BaseAgent, FullTerminalUse}`.
- `app/src/terminal/input/models/view.rs:75` builds the Full Terminal Use tab with `QueryFilter::FullTerminalUseModels`.
- `app/src/terminal/input/models/data_source.rs:149` switches active model and choices based on the tab filter.
- `app/src/terminal/input.rs:4369` writes Base selections to `LLMPreferences::update_preferred_agent_mode_llm`, while Full Terminal Use selections write `AIExecutionProfilesModel::set_cli_agent_model`.

The local OpenAI model is currently only added to `models.agent_mode`:

- `app/src/ai/llms.rs:630` adds custom models from local OpenAI config.
- `app/src/ai/llms.rs:646` makes `Local OpenAI:*` the default `agent_mode` choice.
- `app/src/ai/llms.rs:841` returns `cli_agent` choices when present, falling back to `agent_mode` only when `cli_agent` is absent.

Conclusion: Option A should add a feature-flagged local choice to the Full Terminal Use data source, independent of external CLI-agent choices. Selecting that local namespaced id must not call the external CLI-agent startup path.

## Q2: Panel Reuse Is Feasible

Agent View already owns the panel/chat experience and does not require a `CLIAgentSession`:

- `app/src/terminal/input.rs:14793` handles Start New Agent Conversation by calling `AgentViewController::try_enter_agent_view`.
- `app/src/ai/blocklist/agent_view/controller.rs:687` enters fullscreen agent view unless a long-running command blocks entry.
- `app/src/terminal/view.rs:26187` pins the input to the bottom when Agent View is fullscreen.
- `app/src/terminal/view.rs:26245` renders the alt-screen element when a TUI is active, then still renders the normal input/agent view stack.
- `app/src/terminal/view.rs:26658` overlays the Agent View background while fullscreen.

The existing render path can show Agent View over an active alt-screen/TUI. The risk is not render feasibility; the risk is preserving TUI ownership of stdin. Option A should enter Agent View and focus the agent input, but never route submitted text to `write_to_pty` or CLI rich input.

Conclusion: reuse fullscreen Agent View for V0. Do not add a floating panel or new CLI-agent rich input state.

## Q3: TUI Context

Existing context already supplies cwd and selected text:

- `app/src/ai/blocklist/context_model.rs:419` builds pending context.
- `app/src/ai/blocklist/context_model.rs:445` always attaches `AIAgentContext::Directory`.
- `app/src/ai/blocklist/context_model.rs:510` attaches selected text for user queries.
- `app/src/terminal/view.rs:13524` can read selected terminal text, including alt-screen selection.
- `app/src/terminal/view.rs:18474` snapshots selected text into pending context before asking AI.

There is no existing `AIAgentContext` variant for foreground process name:

- `app/src/ai/agent/mod.rs:1977` lists current context variants.
- `app/src/ai/agent/local/mod.rs:4484` serializes context sections for local OpenAI-compatible messages.
- `app/src/ai/agent/api/convert_to.rs:723` maps context to hosted API input context.

The active block command is available from `TerminalModel::block_list().active_block().command_with_secrets_obfuscated(false)`, and the alt-screen active bit is already available on `TerminalModel`.

Conclusion: add a small context variant for active foreground command/process metadata, or an equivalent local-only context section, populated only for FTU-local Agent entry. Do not attach grid snapshot or last command history in V0.

## Q4: Shell Commands From TUI

The local route already converts `run_shell_command` into Warp's shell action path:

- `app/src/ai/agent/local/tool_card.rs:104` classifies local safe commands and builds the tool card.
- `app/src/ai/blocklist/action_model/execute/shell_command.rs:112` decides autoexecution permissions.
- `app/src/ai/blocklist/action_model/execute/shell_command.rs:267` emits `ShellCommandExecutorEvent::ExecuteCommand`.

This path writes a new Warp command through the shell executor and captures output. It is separate from user keystrokes and can keep approval/audit UI.

Risk: when a foreground TUI owns the terminal, direct execution could be blocked or cancelled by `active_block().is_active_and_long_running()` in `shell_command.rs:248`. That is acceptable for V0 if the implementation keeps the existing bounded cancellation/approval behavior, but tests and e2e must verify at least one safe command path while an alt-screen process is active.

Conclusion: keep `run_shell_command` enabled for local FTU conversations. Never inject command bytes into the foreground TUI stdin. The existing `RequestCommandOutput` path is the only allowed execution path.

## Q5: Conversation History

Base Agent conversations are not backed by `CLIAgentSession`; they are tracked by existing Blocklist AI history:

- `app/src/ai/blocklist/controller.rs:632` creates or continues a conversation for a request.
- `app/src/ai/blocklist/controller.rs:1231` sends custom AI inputs through the same conversation selection logic.
- `app/src/ai/blocklist/history_model.rs` owns live and persisted `AIConversation` state.
- `app/src/ai/active_agent_views_model.rs:85` tracks active Agent View conversations per terminal view.

External CLI agents have separate session state in `CLIAgentSessionsModel`, but Option A does not need to create or spoof that state. Creating a separate FTU-local ID model would duplicate history without adding capability.

Conclusion: reuse existing Base Agent local conversation persistence. Add metadata only if the implementation needs to label the entry origin as Full Terminal Use for tests/telemetry; do not add a new persistence model.

## Overall Decision

Option A is feasible with scoped changes:

1. Add a default-off `LocalAgentFullTerminalUse` feature flag.
2. Add the configured local model to FTU choices as `Local Agent`.
3. When FTU selected id is local, start/reuse Agent View local conversation instead of a CLI-agent session.
4. Attach cwd, selected text, and foreground command/process metadata.
5. Keep shell execution on the existing `RequestCommandOutput` path.
6. Reuse existing conversation persistence and Agent View lifecycle.

Escalation is only needed if implementation discovers Agent View cannot be entered while alt-screen is active, or if shell execution from alt-screen requires byte-stream injection.
