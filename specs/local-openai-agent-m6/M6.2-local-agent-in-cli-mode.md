# M6.2: Local Agent in Full Terminal Use

## Summary

Investigate and add a path for the local OpenAI-compatible agent to be available when the terminal is in Full Terminal Use, meaning a foreground TUI or CLI program owns terminal input/output. This is separate from SSH: local and SSH shell prompts use the Base Agent route, while foreground TUI programs such as `vim`, `top`, `htop`, and `tssh` use the Full Terminal Use route.

## User Story

As a Warp user who configured a local OpenAI-compatible agent, I want Local Agent to remain the default option when a foreground TUI is active, so I do not have to switch to a hosted or external CLI agent to ask questions about the current terminal state.

## Current Understanding

Base Agent uses Warp's chat/tool route and serializes OpenAI-compatible messages to the configured local provider. Full Terminal Use is wired around CLI agents that interact with terminal byte streams and plugin/session events:

- `app/src/terminal/cli_agent.rs` defines known CLI agents and command prefixes.
- `app/src/terminal/cli_agent_sessions/*` tracks active CLI-agent sessions, status, rich input state, and plugin-backed context.
- `app/src/ai/blocklist/agent_view/agent_input_footer/mod.rs` chooses `InlineModelSelectorTab::FullTerminalUse` when an agent controls or is tagged in the active block.
- CLI-agent harnesses under `app/src/ai/agent_sdk/driver/harness/*` start external CLIs and observe terminal/session state.

The local OpenAI-compatible agent is not a CLI process and does not currently speak terminal escape sequences or plugin session protocols. Treating it as a normal CLI agent would require a bridge layer, not just a model selector entry.

## Options

### Option A: Local Agent Panel in Full Terminal Use

When Full Terminal Use is selected and a local provider is configured, offer `Local Agent` as a default option that opens the existing Base Agent local route in a side/rich panel while preserving the foreground TUI's terminal ownership.

Pros:

- Lowest implementation risk.
- Reuses existing local OpenAI-compatible serialization, tools, approvals, and settings.
- Avoids pretending the local provider can directly control terminal byte streams.

Cons:

- It is not a true TUI-controlling CLI agent.
- Follow-up interactions may need explicit UI copy that commands/tool calls run via Warp, not via the foreground program's stdin.

### Option B: Synthetic Local CLI-Agent Bridge

Create a local harness that presents the local OpenAI-compatible provider as a CLI-agent session. Warp would translate rich input prompts into OpenAI messages and translate provider tool calls/results into CLI-agent session events.

Pros:

- Better alignment with Full Terminal Use UI and status model.
- Could eventually support richer terminal-state context.

Cons:

- Requires new protocol glue between OpenAI tool calls and CLI-agent session lifecycle.
- Must define how local tools interact with a foreground TUI that owns stdin/stdout.
- Larger blast radius in session tracking, footer state, notifications, and shared-session behavior.

### Option C: Require an External Local CLI Wrapper

Ask users to run a local CLI program that talks to their OpenAI-compatible endpoint, then let Warp detect it as a custom CLI agent.

Pros:

- Minimal Warp implementation.

Cons:

- Poor user experience for this project.
- Does not use the existing Local OpenAI-compatible Agent settings.
- Not recommended for M6.

## Recommendation

Start with Option A unless follow-up research proves it cannot satisfy the core user workflow. It gives users access to the configured local agent in TUI mode without rewriting the CLI-agent protocol layer.

Before implementation, confirm the desired UX with PM:

- The model selector shows Local Agent as the suggested/default option in Full Terminal Use when `local_openai_*` config is present.
- The foreground TUI keeps terminal focus unless the user explicitly runs a command/tool through Warp.
- The local route can attach relevant terminal context, selected text, and cwd, but does not inject keystrokes into the TUI in V0.

## Acceptance Criteria Draft

1. Add a dedicated feature flag for the M6.2 implementation, default off.
2. With the flag off, Full Terminal Use model selector behavior is unchanged.
3. With the flag on and local provider config present, Local Agent appears as the suggested/default Full Terminal Use option.
4. Selecting Local Agent from Full Terminal Use starts the local OpenAI-compatible route without starting an external CLI process.
5. Existing CLI agents such as Claude, Codex, Gemini, and custom CLI detections remain available and unchanged.
6. If no local provider is configured, Local Agent is hidden or disabled with a clear settings path.
7. The implementation must not send raw TUI bytes, image data, or terminal contents to a hosted provider when Local Agent is selected.
8. Manual dogfood verifies `vim`, `top`, `htop`, and `tssh`/similar foreground TUI cases on `WarpLocal.app`.

## Research Questions

- Can the current footer/model selector launch a Base Agent local conversation while `InlineModelSelectorTab::FullTerminalUse` is active?
- Which terminal context is safe and useful when a foreground TUI is active: cwd, selected text, visible grid snapshot, last command, process name?
- Does opening the local panel conflict with CLI-agent rich input state or active TUI paste/keyboard routing?
- Should Local Agent in Full Terminal Use support `run_shell_command`, or should shell execution be hidden because the foreground TUI owns the terminal?
- How should conversation restore/session history represent a local Full Terminal Use conversation that is not backed by a `CLIAgentSession`?

## Non-Goals

- No terminal byte-stream protocol emulation in the first implementation unless PM explicitly approves the larger scope.
- No SSH-specific behavior; this is about shell mode versus foreground TUI mode.
- No changes to external CLI-agent command detection or plugin installation.
- No wave 2 promotion changes.

## Scale Estimate

Option A: M/L depending on how separable the Full Terminal Use selector is from CLI-agent session startup.

Option B: XL because it requires a synthetic CLI-agent bridge and new lifecycle semantics.
