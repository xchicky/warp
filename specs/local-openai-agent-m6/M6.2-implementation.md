# M6.2: Local Agent in Full Terminal Use Implementation Spec

## Context

M6.2 implements Option A from [M6.2-local-agent-in-cli-mode.md](M6.2-local-agent-in-cli-mode.md): expose the configured Local OpenAI-compatible Agent while Full Terminal Use is active, without building a synthetic CLI-agent byte-stream bridge.

Full Terminal Use currently means the model selector and active-model state use the CLI-agent lane:

- `app/src/terminal/input/models/view.rs:59` defines `InlineModelSelectorTab::FullTerminalUse`.
- `app/src/terminal/input/models/view.rs:75` adds the Full Terminal Use tab with `QueryFilter::FullTerminalUseModels`.
- `app/src/terminal/input/models/data_source.rs:149` chooses Full Terminal Use choices from `LLMPreferences::get_cli_agent_llm_choices`.
- `app/src/terminal/input.rs:4369` stores Full Terminal Use selections as `AIExecutionProfilesModel::set_cli_agent_model`.

The local OpenAI-compatible model is injected into Base Agent choices, not necessarily Full Terminal Use choices:

- `app/src/ai/llms.rs:630` reads `ApiKeys::local_openai_config`.
- `app/src/ai/llms.rs:646` makes the local OpenAI choice the `agent_mode` default.
- `app/src/ai/llms.rs:841` uses dedicated `cli_agent` choices when present, so local `agent_mode` fallback is not guaranteed.

The existing Agent View route can host a local conversation without a `CLIAgentSession`:

- `app/src/terminal/input.rs:14793` starts an Agent View conversation from input.
- `app/src/ai/blocklist/agent_view/controller.rs:687` enters fullscreen Agent View.
- `app/src/terminal/view.rs:26187` pins the input while fullscreen Agent View is active.
- `app/src/terminal/view.rs:26658` applies Agent View chrome over the terminal stack.
- `app/src/ai/blocklist/controller.rs:632` creates/continues an `AIConversation` for requests.

Context and shell execution already exist for the local route:

- `app/src/ai/blocklist/context_model.rs:445` attaches cwd through `AIAgentContext::Directory`.
- `app/src/ai/blocklist/context_model.rs:510` attaches selected text.
- `app/src/ai/agent/local/mod.rs:4484` serializes context into local OpenAI-compatible messages.
- `app/src/ai/blocklist/action_model/execute/shell_command.rs:112` enforces autoexecute permissions.
- `app/src/ai/blocklist/action_model/execute/shell_command.rs:267` executes commands through Warp's shell executor.

The research notes are checked in at [_research/M6.2-option-a-research.md](_research/M6.2-option-a-research.md).

## Proposed Changes

### Feature Flag

Add `FeatureFlag::LocalAgentFullTerminalUse`, default off. Do not promote, demote, or clean up any existing local-agent flags in the implementation PR.

When this flag is off, Full Terminal Use model selection, CLI-agent sessions, Agent View entry, context attachment, and shell execution must remain unchanged.

### Full Terminal Use Selector

Add a feature-flagged local option to the Full Terminal Use model list when `local_openai_api_key`, `local_openai_base_url`, and `local_openai_model` are configured.

Implementation guidance:

- Reuse the existing local id namespace from `local_openai_llm_id`, preserving raw provider model names only inside `LocalDirectConfig`.
- Display the FTU item as `Local Agent` or `Local Agent (<model>)`; avoid making it look like an external CLI agent.
- Make it the active/default Full Terminal Use choice when local config is present and no explicit non-local FTU model is selected.
- Preserve existing external CLI-agent choices and their ordering after the suggested local choice.
- If local config is absent, do not show a broken selectable item. The settings path can remain the existing Manage defaults / Warp Agent settings affordance.

The selector acceptance path needs one new branch: if the selected tab is Full Terminal Use and the chosen id is local OpenAI-compatible, store it as the FTU model but mark it as a local FTU route. It must not start, install, or tag in a CLI-agent session.

### Agent View Bridge

When the user invokes Full Terminal Use with the local FTU model selected, open the existing fullscreen Agent View conversation and send prompts through the local OpenAI-compatible route.

Implementation guidance:

- Reuse `AgentViewController::try_enter_agent_view` and `BlocklistAIController` request flow.
- Keep TUI stdin ownership intact. Do not call `write_to_pty`, CLI rich input submission, bracketed paste, or any `CLIAgentSessionsModel` startup path for local FTU prompts.
- The foreground TUI can remain rendered behind the Agent View chrome; Agent View owns user prompt focus while active.
- Existing external CLI-agent behavior must remain unchanged when the FTU model id is not local.
- If a long-running command blocks fullscreen Agent View entry, keep the existing bounded failure/toast behavior and do not fall back to writing into the TUI.

### Context Attachment

For local FTU prompts, attach exactly the V0 context PM approved:

- cwd through existing `AIAgentContext::Directory`
- selected text through existing `AIAgentContext::SelectedText`
- foreground process/command name from the active long-running block while alt-screen/TUI is active

Do not attach by default in V0:

- full grid snapshot
- last command shell history
- raw TUI screen contents
- arbitrary foreground stdin/stdout bytes

The current `AIAgentContext` enum has no process-name variant, so implementation should add a small explicit context variant such as `ForegroundProcess { command: String, is_alt_screen: bool }`, or an equivalent local-only context section if that is less invasive. If adding a shared variant, update local serialization and hosted API conversion/fallback deliberately so hosted providers do not receive unreviewed local-only TUI data.

The command/process string must be secret-obfuscated, using the same command rendering policy as terminal block display.

### Shell Execution

Local FTU conversations support `run_shell_command`, but only through the existing Warp action path:

- provider emits `run_shell_command`
- local tool card converts it to `AIAgentActionType::RequestCommandOutput`
- Warp renders approval/audit UI
- `ShellCommandExecutor` executes the command outside the foreground TUI stdin path
- command output is returned to the local provider in the follow-up round

This preserves M6.1 safe-command autoexecute: if `LocalAgentShellExecution` and `LocalAgentAutoExecuteSafeCommands` are enabled, safe commands may autoexecute; unsafe commands still require visible approval.

Do not implement byte-stream injection into vim/top/htop/tssh. Do not route commands through CLI rich input or bracketed paste.

### Conversation History

Reuse existing Base Agent local conversation persistence:

- `BlocklistAIHistoryModel` remains the owner of `AIConversation`.
- `AgentViewController` remains the per-terminal active view state.
- `ActiveAgentViewsModel` continues to track active conversations.
- No new FTU-local conversation id namespace is needed.
- No fake `CLIAgentSession` should be created for local FTU conversations.

Add an entry origin or metadata only if needed to make tests, telemetry, or UI labels unambiguous.

## Testing And Validation

Every new code path needs focused tests before e2e verification.

Required unit or model tests:

1. FTU selector: with `LocalAgentFullTerminalUse` off, Full Terminal Use choices are unchanged.
2. FTU selector: with flag on and local OpenAI config present, Local Agent appears in Full Terminal Use choices and is selected/default when no explicit non-local FTU model exists.
3. FTU selector: existing external CLI-agent choices remain available when Local Agent is present.
4. Selection bridge: choosing local FTU model does not create or require a `CLIAgentSession`.
5. Route bridge: local FTU prompt uses `local_direct_config` for the namespaced local id and does not route to hosted OpenAI.
6. Context: local FTU prompt includes cwd, selected text, and foreground process/command metadata.
7. Context boundary: local FTU prompt excludes grid snapshot and last command history by default.
8. Shell execution: local FTU `run_shell_command` becomes `RequestCommandOutput`, not PTY byte injection.
9. M6.1 regression: safe local commands still use `local_autoexecute_safe`; unsafe commands still require approval.
10. External CLI-agent regression: non-local FTU model selection still follows existing CLI-agent behavior.

## Risks And Mitigations

- Selector confusion: Local Agent is not a true CLI-agent. Mitigate with explicit display label and by not creating `CLIAgentSession` state.
- TUI stdin corruption: any path that writes prompt text or command bytes to the foreground TUI is out of scope. Tests must assert the local FTU route does not call CLI rich input or PTY write paths for prompt submission.
- Context over-collection: V0 must not attach grid snapshots or last command history. Tests should check absence, not only presence.
- Shell execution while TUI is active: existing executor may cancel when another long-running command is active. Preserve bounded behavior; do not work around it by injecting bytes into the TUI.
- Local model id collisions: keep `local-openai:<model>` namespacing through selector and routing.

## Non-Goals

- No synthetic CLI-agent bridge.
- No byte-stream protocol emulation.
- No foreground TUI keystroke injection.
- No new floating panel.
- No broad Agent View or footer redesign.
- No wave promotion or rollout changes.
- No grid snapshot or shell history context by default.

## Parallelization

Implementation should be a single PR owned by one agent. The changed paths are tightly coupled: selector state, Agent View entry, context construction, and shell execution tests must agree on the same local FTU route marker. Parallel sub-agents would create merge churn without much wall-clock benefit. Agent2 should review the implementation PR before agent1 merges.
