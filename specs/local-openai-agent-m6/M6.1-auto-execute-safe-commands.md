# M6.1: Local Agent Auto-Execute Safe Commands

## Summary

Add a default-off feature flag that lets the local OpenAI-compatible agent auto-execute a fixed set of read-only shell commands. The existing `run_shell_command` tool card and command output history remain visible; the user only skips the approval click for commands that pass the local static classifier.

## User Story

As a Warp user running the local OpenAI-compatible agent, I want harmless inspection commands to run without repeated manual approval, so the agent can complete multi-step debugging and repository inspection tasks with less interruption.

## Feature Flag

Add `FeatureFlag::LocalAgentAutoExecuteSafeCommands`, default off. Do not promote, demote, or clean up any existing local-agent flags in the implementation PR.

## V0 Behavior

When the flag is enabled and `FeatureFlag::LocalAgentShellExecution` is also enabled:

1. A `run_shell_command` request is classified locally before approval.
2. If the full command matches the static safe-command classifier, Warp auto-executes it through the existing shell command action/executor path.
3. If the command does not match, the existing M1.3 visible approval path is unchanged.
4. The command card, terminal output, command result card, local-provider resume/finalize round, and audit history remain visible.
5. Plan mode remains read-only and must not auto-execute shell commands.
6. Subagents must not receive this capability in V0 unless a later spec explicitly expands child shell execution.

The implementation must not trust model-provided `is_read_only` or `is_risky` arguments. The classifier result is the only V0 source of auto-execute eligibility.

## Static Allowlist

The V0 allowlist is complete. Commands not listed here require approval.

Allowed top-level commands:

- `pwd`
- `ls`
- `cat`
- `head`
- `tail`
- `wc`
- `grep`
- `rg`
- `find`
- `echo`
- `which`
- `type`
- `command -v`
- `ps`
- `df`
- `du`
- `file`
- `uname`
- `date`
- `whoami`
- `id`
- `stat`
- `tree`

Allowed `git` subcommands:

- `git status`
- `git diff`
- `git log`
- `git show`
- `git rev-parse`
- `git ls-files`
- `git grep`

Allowed bounded `top` forms:

- macOS: `top -l <positive-integer>` with any other read-only display flags.
- Linux-style batch: `top -b -n <positive-integer>` with any other read-only display flags.

Bare `top` is not auto-executed because it can take over the terminal indefinitely.

## Fail-Closed Parsing Rules

The classifier operates on the entire shell command string. It must fail closed and keep the existing approval flow when any of these are present outside inert single-quoted text:

- command chaining or separators: `;`, newline, `&&`, `||`
- pipelines or backgrounding: `|`, `&`
- redirection: `>`, `>>`, `<`, `<<`, `<<<`, `2>`, `&>`
- command substitution: `$(`, backticks
- process substitution: `<(`, `>(`
- shell evaluation wrappers: `eval`, `exec`, `source`, `.`, `sh`, `bash`, `zsh`, `fish`, `pwsh`
- remote/network/file-transfer commands: `ssh`, `scp`, `rsync`, `curl`, `wget`, `nc`, `netcat`, `telnet`
- delete/write/package commands: `rm`, `mv`, `cp`, `touch`, `mkdir`, `rmdir`, `chmod`, `chown`, `sudo`, `tee`, package managers, editors

Variable expansion such as `echo $HOME` is allowed when the command otherwise matches the allowlist. Command substitution inside variables is not allowed because `$(` and backticks fail closed.

The parser should use an existing shell lexer/parser if one is already practical in the codebase. If implementation uses a small purpose-built parser, it must be covered with table tests and must not attempt partial matching. `ls && rm -rf /` and `ls; rm -rf /` are not "safe `ls`"; the whole command requires approval.

## Command-Specific Deny Rules

Even when the top-level command is allowlisted, these argument patterns require approval:

- `find`: `-delete`, `-exec`, `-execdir`, `-ok`, `-okdir`, `-fprint`, `-fprintf`, `-fls`
- `grep` / `rg`: options that write files or run preprocessors, including `--output`, `--pre`, `--pre-glob` when paired with executable preprocessors
- `git diff` / `git show` / `git log`: `--output`, `--ext-diff`, `--external-diff`, `--exec-path`, `-c`, `--config-env`
- `du`: no deletion flags exist, but commands with redirection or chaining are still rejected by the global parser
- `echo`: allowed only as a single command; shell redirection or command substitution still rejects

If implementation cannot robustly classify a command-specific option, it must require approval.

## Acceptance Criteria

1. `LocalAgentAutoExecuteSafeCommands` is added to `crates/warp_features/src/lib.rs` and re-exported through the normal runtime feature bridge.
2. With the new flag off, `run_shell_command` behavior is byte-for-byte equivalent for approval behavior: valid commands wait for visible approval.
3. With the flag on, an allowlisted command such as `pwd`, `ls`, `cat Cargo.toml`, `rg LocalAgent`, or `git status --short` auto-executes through the existing shell action/executor path.
4. Auto-executed commands still render command execution UI and return bounded command output to the local provider.
5. Non-allowlisted commands such as `cargo test`, `python script.py`, `npm install`, `rm file`, or `sed -i ...` still require visible approval.
6. Injection attempts such as `ls && rm -rf /`, `cat Cargo.toml; rm x`, ``echo `rm x` ``, `echo $(rm x)`, `grep foo file > out`, and `find . -delete` require visible approval.
7. The classifier ignores model-provided `is_read_only` and `is_risky` when deciding auto-execution.
8. Existing user denylist settings continue to take precedence over auto-execute.
9. Existing user-visible approvals for denied/fallback commands, cancellation results, action-result resume, and local-provider finalize remain unchanged.
10. No shell command is auto-executed in plan mode.

## Tests

Add focused tests for:

- every allowlisted command form above
- every forbidden shell operator above
- each command-specific deny rule
- model-claimed `is_read_only: true` on unsafe commands still requiring approval
- feature-flag off fallback
- denylist precedence
- local route `run_shell_command` action/card path still returning visible command output and final local-provider response

Manual dogfood verification on `WarpLocal.app`:

1. Configure Local OpenAI-compatible Agent.
2. Ask the agent to inspect the current `warp` repository with several read-only shell steps.
3. Confirm allowlisted commands run without approval clicks and command cards remain visible.
4. Ask for a non-allowlisted command such as `cargo test -p warp ai::agent::local:: -- --nocapture`; confirm approval is still required.
5. Try an injection prompt that asks for `ls && rm -rf /`; confirm approval is required and nothing auto-executes.

## Non-Goals

- No model self-certification.
- No restoration of M1.3 `is_read_only` / `is_risky` trust.
- No path containment checks for command arguments in V0.
- No auto-execution for file writes, MCP tools, web tools, subagents, or plan mode.
- No changes to cloud agent command execution policy.

## Implementation Notes

Recommended implementation path:

1. Add a pure classifier near the local shell execution path or shared command-permission layer.
2. Reuse the existing `RunShellCommand` tool card / `AIAgentActionType::RequestCommandOutput` / `ShellCommandExecutor` path so terminal execution, output capture, denylist behavior, resume/finalize, and audit UI stay consistent.
3. Add a new permission reason or local-only branch so telemetry can distinguish static local safe-command auto-execution from model-decided read-only execution.
4. Keep mutating/shell command batches sequential.

If implementation finds the static allowlist cannot be made reliable with the current shell parsing utilities, stop and escalate before broadening trust.
