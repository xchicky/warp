# M2.1: Local Agent MCP Client Lifecycle and Configuration

## Summary

Add the feature-flagged plumbing for the local OpenAI-compatible agent to see MCP servers that Warp has already configured, started, and scoped to the current context. M2.1 should not expose MCP tools to the provider yet; it establishes the safe source of truth and lifecycle contract that later specs consume.

Current branch note: the local direct agent currently lives under `app/src/ai/agent/local/*`. Existing MCP managers live under `app/src/ai/mcp/*`.

## User Story

As a Warp user with MCP servers configured in Warp, I want my local OpenAI-compatible agent to use the same MCP server setup as the rest of Warp, so I do not need to duplicate server configuration for local providers.

## Acceptance Criteria

1. Add `FeatureFlag::LocalAgentMcp`, default off, following the M1 feature flag pattern:
   - `crates/warp_features/src/lib.rs`
   - `crates/warp_core/src/features.rs` if needed by the local feature bridge
   - `app/Cargo.toml` compile feature
   - `app/src/lib.rs` runtime bridge
2. When `LocalAgentMcp` is off, local OpenAI-compatible requests behave exactly as they do after M1. No MCP server discovery, tool injection, server start, or MCP-specific history conversion runs from the local-agent path.
3. When `LocalAgentMcp` is on, the local-agent path can query active MCP server metadata from Warp's existing MCP managers without starting an independent stdio process runner inside `app/src/ai/agent/local`.
4. Configuration source is existing Warp MCP state:
   - Settings -> MCP Servers and cloud-synced templatable installations.
   - File-based MCP installations already discovered by `FileBasedMCPManager`.
   - Project-scoped config that applies to the current terminal cwd.
   - Global config only when Warp's existing MCP discovery already treats it as active or runnable.
5. M2.1 must not add a new local-agent-only MCP config file or a new local-agent-only config parser.
6. M2.1 starts with stdio/CLI MCP servers. SSE, streamable HTTP, and OAuth-specific expansion are out of scope unless the existing manager already has an active connected server that later sub-milestones can safely consume without new transport work.
7. The active server set is scoped the same way the regular agent request path scopes MCP context:
   - Active templatable servers.
   - Active file-based servers for the current cwd when a cwd is available.
   - Active CLI-spawned servers if the existing manager exposes them.
8. If no cwd is available, local-agent MCP discovery may include only globally active servers; project-scoped file servers must not be guessed.
9. If no MCP servers are active or discoverable, the local agent still advertises all non-MCP local tools normally.
10. Server startup state is not hidden from the user. If a configured server is stopped, starting, authenticating, or failed, M2.1 exposes enough internal API for later milestones to either skip it or produce a clear local diagnostic.
11. Existing MCP manager ownership remains unchanged. `TemplatableMCPServerManager` and `FileBasedMCPManager` remain the source of lifecycle truth; local-agent code should not retain long-lived server process handles.
12. Existing 74 local-agent tests continue to pass. Add focused tests for flag gating, empty MCP context, cwd-scoped server selection, and "no independent local stdio runner" behavior where practical.

## Safety Risks

- Duplicate process launch: a local-agent-only MCP runner could start the same external process twice. Mitigate by consuming existing Warp managers.
- Wrong scope: a project MCP server from another repo could be exposed to the current repo. Mitigate by reusing `FileBasedMCPManager` cwd scoping and rejecting guessed roots.
- Secret leakage: MCP configs may contain env placeholders or secrets. Mitigate by never echoing raw resolved config into provider messages.
- Surprise enablement: users may not expect MCP tools when the flag is off. Mitigate with `LocalAgentMcp` default off and no side effects when disabled.

## Scale Estimate

M. Most lifecycle code exists, but the local direct agent currently does not carry an MCP context, so this requires careful interface design and tests around scoping.

## PR Split Suggestion

Keep M2.1 as one PR with at most five commits:

1. Add `LocalAgentMcp` feature flag, default off.
2. Add a local-agent MCP context helper that reads active server metadata from existing MCP managers.
3. Add cwd/global scoping tests with test doubles or lightweight manager fixtures.
4. Add no-op integration into `generate_openai_compatible_output` that preserves M1 behavior while the flag is off.
5. Add documentation comments and test fixtures for later M2.2/M2.3 use.

Each commit should compile independently and avoid changing the existing non-local MCP execution path.

## Entry Points

- `app/src/ai/agent/api.rs`: existing `RequestParams::new` builds `MCPContext`; use this as the model for what server metadata local direct should see.
- `app/src/ai/agent/local/mod.rs`: thread optional MCP context into local tool construction without changing M1 tools.
- `app/src/ai/mcp/templatable_manager.rs`: use active server metadata and `TemplatableMCPServerInfo`.
- `app/src/ai/mcp/file_based_manager.rs`: rely on existing cwd/project scoping.
- `app/src/lib.rs`, `app/Cargo.toml`, `crates/warp_features/src/lib.rs`: add feature flag plumbing.
