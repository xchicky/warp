# M2.3: Local Agent MCP Tool Invocation Passthrough

## Summary

Route local provider calls to namespaced MCP tools through Warp's existing MCP action and approval path. The local-agent code should not call MCP peers directly in an unreviewed background task when an existing `CallMCPTool` action path can preserve user confirmation, permissions, telemetry, reconnection, and UI consistency.

## User Story

As a Warp user, I want the local agent to call an MCP tool only after I approve the visible request, and then use the tool result to continue the same local conversation.

## Acceptance Criteria

1. When the local provider calls a namespaced MCP tool from M2.2, the local tool runner validates:
   - `LocalAgentMcp` is enabled.
   - The tool name maps to an active MCP server/tool from the current catalog.
   - Arguments are valid JSON object arguments for that tool.
2. MCP tool invocation is out-of-band by default, matching M1.3 shell execution:
   - Emit a structured `Message::ToolCall` using existing `CallMcpTool` proto shape where practical.
   - Finish the current stream while the user approval/execution path owns the external action.
   - On `ActionResult` / resume, convert the MCP result back into an OpenAI `tool` message and run the local finalize round.
3. Every MCP tool call requires visible user approval by default in M2. The confirmation must show:
   - Server name.
   - Tool name.
   - Namespaced local function name.
   - Bounded formatted arguments.
   - A clear indication that the call goes to an external user-configured MCP server.
4. Rejection or cancellation returns a bounded permission-denied/cancelled tool result to the local provider. It must not be treated as provider transport failure.
5. Approval forwards the original MCP tool name and JSON object arguments to the existing MCP execution path. The local provider's namespaced function name must not be sent to the MCP server as the tool name.
6. Arguments are treated as opaque JSON from the model. The implementation must not compose shell commands or file paths from fragments beyond the MCP server's own schema.
7. Provider-facing results include:
   - Server name and tool name.
   - `status`: success, error, cancelled, timeout, or unavailable.
   - Text content and structured content when present, bounded by byte and character limits.
   - An explicit truncation marker when output is truncated.
8. Binary, image, and resource content returned by MCP tools are not sent unbounded to the text-only local provider in M2. They are represented as bounded metadata or a clear unsupported-content note.
9. MCP tool results are appended to local OpenAI-compatible message history with the same tool call id the provider used, so the provider can continue/finalize normally.
10. History conversion handles MCP `Message::ToolCall` and `ToolCallResult` in `append_history_messages`, just as M1.3 handles `RunShellCommand`.
11. If multiple MCP tools or other mutating tools are emitted in one provider batch, execute/pause in provider order. Do not approve or dispatch later MCP/mutating calls until the earlier out-of-band action returns.
12. Existing local tools, shell execution, file writes, and the 74 local-agent tests continue to pass. Add tests for valid MCP pending action, approval result conversion, rejection conversion, namespaced-to-MCP-name routing, sequential scheduling, and result truncation.

## Safety Risks

- Unauthorized external action: an MCP tool can modify external services. Mitigate with visible approval for every call by default.
- Argument disclosure: arguments can contain secrets from provider context. Mitigate by bounding UI display and never logging raw full arguments in safe logs.
- Wrong server routing: duplicate tool names can call the wrong integration. Mitigate with M2.2 namespacing and catalog lookup by server id.
- Provider context blowup: MCP results can be huge. Mitigate by byte/char truncation and unsupported-content markers.
- UI inconsistency: raw local text would hide the approval boundary. Mitigate by using structured `CallMcpTool` cards/results where practical.

## Scale Estimate

L. This crosses local streaming, structured tool cards, user approval, existing MCP executor, action result resume, and result shaping.

## PR Split Suggestion

Keep M2.3 as one PR after M2.2, with at most six commits:

1. Add namespaced MCP dispatch validation and tool-call mapping.
2. Emit structured `CallMcpTool` pending action events from the local tool runner.
3. Convert MCP `ActionResult` / `ToolCallResult` back into OpenAI-compatible tool messages.
4. Add provider-facing MCP result formatting and truncation.
5. Add ordered scheduling tests for mixed MCP/local mutating batches.
6. Add structured card/YAML consistency tests if the proto mapping requires local `tool_card.rs` changes.

## Entry Points

- `app/src/ai/agent/local/mod.rs`: extend `execute_local_tool`, pending tool-call handling, history conversion, and result formatting.
- `app/src/ai/agent/local/tool_card.rs`: add `CallMcpTool` call/result mapping if existing structured card code does not already cover it.
- `crates/ai/src/agent/action/convert.rs`: reuse existing `CallMcpTool` conversion instead of inventing a new action type.
- `app/src/ai/blocklist/action_model/execute/call_mcp_tool.rs`: preserve existing approval/execution behavior.
- `app/src/ai/agent/api/convert_conversation.rs`: reuse existing MCP action result conversion where possible.
