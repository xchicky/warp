# M2.4: Local Agent MCP Errors, Timeout, Reconnect, and Hardening

## Summary

Harden local-agent MCP passthrough so server crashes, stale catalogs, malformed outputs, slow calls, and reconnect attempts produce bounded tool results instead of hangs, panics, or remote-route fallbacks.

## User Story

As a Warp user, I want MCP failures to be understandable and contained, so one broken integration does not derail my local agent session or hide what happened.

## Acceptance Criteria

1. MCP startup, discovery, and invocation failures produce clear provider-facing tool results. The local provider receives a tool result for expected MCP failures, not a stream-level provider error, unless the local provider request itself failed.
2. A stale namespaced tool call, where the tool was advertised earlier but the server/tool is no longer active, returns `status: unavailable` with server/tool identity when known.
3. Transport-closed failures reuse existing `ReconnectingPeer` behavior where possible:
   - Attempt at most one reconnect/retry for a tool call.
   - If reconnect fails, return a bounded `unavailable` or `transport-error` tool result.
   - Do not loop indefinitely.
4. Tool calls have a conservative timeout. If the existing MCP executor already enforces one, document and test it. If it does not, add a bounded local-agent MCP timeout around the out-of-band wait/result path without killing unrelated MCP server state.
5. Timeout results clearly state that the MCP call timed out and include any safely available partial or diagnostic output. The provider can continue/finalize after a timeout.
6. Server stderr/stdout diagnostics are not streamed into provider context unless they are part of the MCP tool result. Safe logs may include summarized failure causes; full logs must not leak secrets.
7. Malformed MCP results, unsupported content blocks, or non-UTF-8/binary content are converted into bounded unsupported-content notes instead of panicking.
8. Output truncation uses both byte and character limits and preserves the tail by default when the result is text. Structured JSON results may preserve the top-level status plus a tail-truncated serialized representation.
9. User-visible tool cards show success, cancelled, timeout, unavailable, and server-error states distinctly enough for debugging.
10. Local conversation routing remains local on `ActionResult` and `ResumeConversation` for MCP calls. It must not fall back to the remote server because the conversation id was generated locally.
11. MCP errors do not poison the local tool loop. After an MCP error result, the provider may call other local tools in a later round up to the existing local tool-round limit.
12. Existing MCP behavior for non-local agents remains unchanged unless a shared helper bug is fixed with tests.
13. Existing 74 local-agent tests continue to pass. Add tests for stale catalog, missing server, transport error, reconnect failure, timeout shaping, unsupported content, truncation, and local-route resume.

## Safety Risks

- Hanging sessions: stdio MCP servers can stop responding. Mitigate with timeout and bounded cancellation behavior.
- Reconnect storms: repeated retries can spawn processes or consume resources. Mitigate with one retry per tool call and existing manager ownership.
- Secret exposure in diagnostics: server errors can include credentials or arguments. Mitigate with safe logs and bounded provider text.
- Broken server blast radius: one server failure can interrupt unrelated local tools. Mitigate by returning tool-level errors and keeping the local tool loop alive.
- Stale approval: user might approve a visible call after the server identity changed. Mitigate by resolving server id/tool identity at execution time and failing closed if the mapping changed.

## Scale Estimate

M/L. Much of the reconnect and action result machinery exists, but timeout and output shaping need focused tests across async boundaries.

## PR Split Suggestion

Keep M2.4 as one PR after M2.3, with at most six commits:

1. Add MCP provider-facing result status enum/helper for success, error, cancelled, unavailable, timeout, and unsupported content.
2. Add stale catalog and missing server handling tests.
3. Add timeout handling or document/reuse existing timeout behavior with tests.
4. Add reconnect failure shaping around existing `ReconnectingPeer`.
5. Add output truncation and unsupported content tests.
6. Add local-route resume regression tests for MCP `ActionResult` and `ToolCallResult`.

## Entry Points

- `app/src/ai/agent/local/mod.rs`: MCP result formatting, truncation, route-resume handling, and fallback behavior.
- `app/src/ai/blocklist/action_model/execute/call_mcp_tool.rs`: existing MCP execution result conversion and reconnect boundaries.
- `app/src/ai/mcp/reconnecting_peer.rs`: one-retry reconnect behavior.
- `crates/ai/src/agent/action_result/mod.rs`: `CallMCPToolResult` display/status shaping.
- `app/src/ai/agent/api/convert_conversation.rs`: MCP action result conversion for history and resume.
