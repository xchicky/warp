# M2.2: Local Agent MCP Tool Discovery and Schema Injection

## Summary

Expose active MCP server tools to the local OpenAI-compatible provider as OpenAI function tools. Tool names are namespaced, schemas are converted from MCP JSON Schema, and unsupported tools are skipped with diagnostics rather than advertised in a broken form.

## User Story

As a Warp user with an active MCP server, I want the local agent to know which MCP tools it can request, so it can use project-specific integrations without me copying tool descriptions into the prompt.

## Acceptance Criteria

1. MCP tools are advertised only when `FeatureFlag::LocalAgentMcp` is enabled and M2.1 can provide at least one active server with at least one usable tool.
2. Existing local tools keep their current names and behavior. MCP tool names must never collide with local tool names such as `read_file`, `grep`, `apply_file_diff`, `run_shell_command`, or future local tools.
3. Each MCP function name uses the deterministic namespace:
   - `mcp__<server_slug>__<tool_slug>`
   - Slugs are lowercase ASCII, preserve alphanumeric characters, convert other runs to `_`, trim leading/trailing `_`, and use a stable fallback derived from the server id or tool index when empty.
4. If two active servers or tools produce the same namespaced function name, the later duplicate is disambiguated with a stable short suffix derived from the server id. No MCP tool is silently allowed to shadow another exposed tool.
5. Each MCP tool definition includes a concise description containing the server display name and the MCP tool description when provided.
6. MCP input schemas are converted into OpenAI function parameters:
   - Preserve `type`, `properties`, `required`, `description`, `enum`, `items`, numeric bounds, string bounds, and `additionalProperties` when representable.
   - Treat a missing schema as an empty object schema.
   - Skip tools whose schema cannot be represented as a JSON object schema.
7. Unsupported schema constructs must not panic or produce invalid OpenAI tool definitions. The tool is skipped and a bounded diagnostic is logged.
8. The provider does not receive raw server config, env vars, resolved secrets, process command lines, or OAuth credentials as part of tool descriptions or schemas.
9. Tool discovery is refreshed for each local provider request or tool round using the latest active server metadata. The local agent does not cache a stale catalog across unrelated conversations.
10. If a server disconnects after tools were advertised but before invocation, M2.3/M2.4 handle the invocation failure; M2.2 only guarantees catalog correctness at the time the request is built.
11. MCP resources are out of scope for M2.2. Do not expose `read_mcp_resource` as an OpenAI function unless a later spec explicitly adds resources.
12. Existing 74 local-agent tests continue to pass. Add tests for flag gating, namespacing, duplicate disambiguation, local-tool collision prevention, schema conversion, schema skip behavior, and secret redaction in descriptions.

## Safety Risks

- Tool collision: an MCP tool named `read_file` could trick the model into using a different behavior. Mitigate with required `mcp__...` namespace.
- Prompt leakage: server config can contain secrets. Mitigate by exposing only server name, tool name, description, and input schema.
- Invalid schema injection: malformed MCP schemas can break provider requests. Mitigate by validating before advertisement and skipping unsafe tools.
- Catalog drift: server tools can change after reconnect. Mitigate by refreshing catalog per request/round and treating invocation lookup as authoritative.

## Scale Estimate

M. The work is mostly schema conversion and naming, but correctness depends on stable mapping from provider tool names back to MCP server/tool identities.

## PR Split Suggestion

Keep M2.2 as one PR after M2.1, with at most five commits:

1. Add MCP tool-name slugging and collision tests.
2. Add MCP JSON Schema to OpenAI function schema conversion with unit tests.
3. Extend `local_tools()` to append MCP tools behind `LocalAgentMcp`.
4. Add catalog mapping from namespaced OpenAI name back to server id and MCP tool name.
5. Add regression tests that existing local tools and M1 flags remain unchanged.

## Entry Points

- `app/src/ai/agent/local/mod.rs`: extend `local_tools()` to accept MCP catalog input and append namespaced tools.
- `app/src/ai/agent/api/convert_to.rs`: reuse or mirror existing `convert_mcp_tool` schema conversion where possible.
- `app/src/ai/agent/mod.rs`: use `MCPContext` / `MCPServer` shapes as catalog inputs.
- `app/src/ai/mcp/templatable_manager.rs`: source server id, server name, and `rmcp::model::Tool` metadata.
