# M1.2: Local OpenAI-Compatible Agent `write_file` and `edit_file`

## Summary

Add feature-flagged file-writing primitives that complement `apply_file_diff`: `write_file` for full-file creation/replacement and `edit_file` for small exact replacements. These tools give the local agent simpler options for low-risk edits where a unified diff is unnecessarily heavyweight.

Current branch note: PM referenced `crates/ai/src/agent/local/*`; this branch's local direct agent currently lives under `app/src/ai/agent/local/*`. If the module moves back under `crates/ai`, use the same entry points there.

## User Story

As a Warp user running a local OpenAI-compatible coding agent, I want the agent to write a new small file or make an exact localized edit, so that common tasks like adding a config file or replacing a function body do not require fragile patch formatting.

## Acceptance Criteria

1. `write_file` and `edit_file` are advertised only when `FeatureFlag::LocalAgentFileWrites` is enabled. The same flag from M1.1 gates all local file mutation.
2. If the flag is off and the provider still calls either tool, the client returns a tool error and performs no filesystem writes.
3. `write_file` accepts at minimum:
   - `path`: absolute path or path relative to the current terminal cwd.
   - `content`: UTF-8 text to write.
   - `overwrite`: optional boolean, default `false`.
4. `write_file` creates a new UTF-8 text file when the target does not exist and the parent directory already exists under the writable root.
5. `write_file` refuses to overwrite an existing file unless `overwrite` is `true`. When it refuses, the error tells the model to use `edit_file`, `apply_file_diff`, or retry with `overwrite: true` only if replacement is intended.
6. `write_file` does not create missing parent directories in M1.2 unless explicitly added to the schema with a default-off option. This avoids accidental broad filesystem changes in the minimum PR.
7. `edit_file` accepts at minimum:
   - `path`: absolute path or path relative to the current terminal cwd.
   - `old_text`: exact UTF-8 text to replace.
   - `new_text`: replacement UTF-8 text.
   - `replace_all`: optional boolean, default `false`.
8. `edit_file` reads an existing UTF-8 text file, finds `old_text`, and writes the result back only when the match rules are unambiguous:
   - zero matches -> tool error, no write.
   - one match -> replace it.
   - more than one match with `replace_all: false` -> tool error, no write.
   - more than one match with `replace_all: true` -> replace all matches.
9. `edit_file` rejects empty `old_text` to avoid unbounded insertion behavior.
10. Both tools share M1.1 writable-root validation: canonicalize the root, canonicalize existing targets or parent directories, reject symlink escapes, and reject paths outside the approved root.
11. Both tools reject binary/non-UTF-8 content and enforce bounded file sizes and bounded input payload sizes.
12. Writes are atomic per target where the platform allows it: write to a temporary sibling file and rename over the target. If the final rename fails, return an error and avoid leaving a partial target file.
13. The result returned to the provider includes a concise summary: path, created/updated status, bytes written or replacements made, and line additions/removals when cheap to compute. Full file contents are not echoed.
14. Error paths are clear and testable: invalid JSON, missing required fields, empty path, no cwd for relative paths, outside writable root, missing parent, existing file without overwrite, missing file for edit, zero/multiple matches, empty `old_text`, oversized file/content, non-UTF-8 file, and I/O failure.
15. Mutating tools are executed in provider order, not in parallel. This applies when `write_file` or `edit_file` appears with any other mutating tool in the same provider tool batch.
16. Existing read-only tools, `apply_file_diff`, and the existing 44 unit tests continue to pass. Add unit tests for create, overwrite refusal, explicit overwrite, exact edit, multiple-match refusal, replace-all, path escapes, size limits, flag gating, and sequential scheduling.

## Safety Risks

- Overwrite/data loss: `write_file` can replace entire files. Mitigate with `overwrite: false` by default and clear errors.
- Ambiguous edits: exact replacements can affect the wrong occurrence. Mitigate with multiple-match refusal unless `replace_all` is explicit.
- Path escape and symlink attacks: reuse M1.1 writable-root checks and reject symlink escapes.
- Partial files: failed writes can corrupt files. Mitigate with temporary sibling writes and rename.
- Resource abuse: large content payloads can exhaust memory. Mitigate with max content bytes and max file bytes.

## Scale Estimate

M. The schemas and executors are straightforward, but correctness depends on shared safety helpers, atomic writes, and broad unit coverage.

## PR Split Suggestion

Prefer one PR after M1.1 lands, with at most five commits:

1. Reuse or extract M1.1 writable-root and mutating-tool scheduling helpers.
2. Add `write_file` schema, executor, and tests.
3. Add `edit_file` schema, executor, and tests.
4. Add structured card/result rendering and local prompt updates.
5. Add regression tests covering mixed mutating batches and flag-off behavior.

If M1.1 has not landed, do not duplicate the safety layer; either stack on that branch or land a shared safety-helper commit first.
