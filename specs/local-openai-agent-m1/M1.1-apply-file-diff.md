# M1.1: Local OpenAI-Compatible Agent `apply_file_diff`

## Summary

Add a feature-flagged mutating local tool that applies a unified diff patch to files under the active local workspace so the local OpenAI-compatible agent can make code changes without asking the user to copy patches manually.

Current branch note: PM referenced `crates/ai/src/agent/local/*`; this branch's local direct agent currently lives under `app/src/ai/agent/local/*`. If the module moves back under `crates/ai`, use the same entry points there.

## User Story

As a Warp user running a local OpenAI-compatible coding agent, I want the agent to apply a proposed patch to files in my current project, so that it can complete small code-change tasks end to end while still staying inside explicit local-write guardrails.

## Acceptance Criteria

1. The tool is advertised to the provider only when `FeatureFlag::LocalAgentFileWrites` is enabled. The flag is added in `crates/warp_features/src/lib.rs`, is re-exported through `crates/warp_core/src/features.rs`, and is off by default in all default flag lists.
2. If the flag is off and the provider still sends `apply_file_diff`, the client returns a tool error such as `Tool error: apply_file_diff is disabled by feature flag` and does not touch the filesystem.
3. The tool schema is named `apply_file_diff` and accepts at minimum:
   - `patch`: unified diff text.
   - `summary`: optional short description for UI/log output.
4. The tool accepts standard unified diff headers for existing file updates, including `diff --git`, `---`, `+++`, and one or more `@@` hunks. Paths may be repository-relative or cwd-relative.
5. The M1.1 minimum viable behavior may reject creates, deletes, renames, mode changes, binary patches, and non-text patches with clear tool errors. If supported in M1.1, they must obey the same path and safety rules as updates.
6. Every touched path resolves under the active writable root before any file is written. For M1.1, the writable root is the current terminal cwd returned by `local_tool_cwd`; if no cwd is available, relative paths are rejected and absolute paths must still be under an approved root.
7. Path validation canonicalizes the writable root and the existing target file. For non-existing targets, it canonicalizes the parent directory. Writes through symlink targets or symlink parent escapes are rejected unless the final canonical target remains under the writable root.
8. The patch applies atomically at the tool level: if any file or hunk fails validation or matching, no files are modified.
9. Existing files must be valid UTF-8 and below a bounded size limit before patching. Use a conservative local constant near the existing `MAX_LOCAL_DIRECT_TOOL_FILE_BYTES`; do not allow unbounded reads or writes.
10. Hunk matching is strict for M1.1: context and removed lines must match the current file content. Fuzzy matching is out of scope unless agent1 can reuse existing diff application helpers without expanding the PR.
11. The result returned to the provider includes a concise success summary: changed file count, per-file line additions/removals, and a bounded list of paths. It must not echo full large file contents.
12. Failure results are actionable and stable: invalid JSON, missing patch, malformed diff, path outside writable root, unreadable file, binary/non-UTF-8 file, hunk mismatch, oversized file, and I/O errors each produce a clear `Tool error: ...` message.
13. The user-visible tool card uses the existing structured `Message::ToolCall` / `ToolCallResult` path where practical. Prefer mapping to `api::message::tool_call::Tool::ApplyFileDiffs` / `api::message::tool_call_result::Result::ApplyFileDiffs` for UI consistency; if unified diff cannot be represented cleanly, render a bounded generic local tool card rather than raw agent text.
14. Mutating tools are not executed in parallel. When a tool batch contains `apply_file_diff`, `write_file`, `edit_file`, or `run_shell_command`, execute the batch in provider index order instead of the current all-`join_all` path.
15. Existing read-only local tools and the existing 44 unit tests continue to pass. Add focused unit tests for tool advertisement gating, path validation, successful patch application, atomic failure, malformed diff, hunk mismatch, and sequential mutating-tool scheduling.

## Safety Risks

- Write-file escape: a model can attempt absolute paths, `..`, symlink traversal, or patch headers that refer outside the project. Mitigate with canonical root checks before opening or writing any file.
- Partial writes: multi-file patches can leave the repo half-mutated. Mitigate by validating all hunks against in-memory file contents before writing any target, then writing only after validation succeeds.
- Data loss: malformed hunks or stale files can overwrite user changes. Mitigate with strict context matching, bounded errors, and no fuzzy matching in M1.1.
- Resource abuse: huge patches or files can exhaust memory. Mitigate with max patch bytes, max files, max hunks, max file bytes, and bounded result text.
- Parallel side effects: the current tool loop runs tools concurrently. Mutating tools must run sequentially in model order.

## Scale Estimate

M. The core is one local tool plus parser/application helpers and tests, but atomic multi-file behavior and path safety make it larger than a simple tool registration.

## PR Split Suggestion

Keep M1.1 as one PR with at most five commits:

1. Add `LocalAgentFileWrites` feature flag, default off.
2. Add unified-diff parser/application helper with unit tests for update-only patches and atomic failures.
3. Wire `apply_file_diff` into local tool schema, flag gating, dispatcher, and sequential mutating-tool scheduling.
4. Add structured card/result rendering for `apply_file_diff`.
5. Add end-to-end local-agent unit coverage and update the local system prompt.

Each commit should compile independently and be revertable without breaking read-only tools.

## `ai::agent::local` Entry Points

- `app/src/ai/agent/local/mod.rs`: rename or split `local_read_only_tools()` into a flag-aware `local_tools()` that adds `apply_file_diff` only when `LocalAgentFileWrites` is enabled.
- `app/src/ai/agent/local/mod.rs`: extend `execute_local_tool()` dispatch with `execute_apply_file_diff_tool(...)`.
- `app/src/ai/agent/local/mod.rs`: add `LocalToolSafety` helpers for writable-root resolution, symlink escape checks, size limits, and mutating-tool classification.
- `app/src/ai/agent/local/mod.rs`: replace unconditional `join_all` execution with a scheduler that preserves parallelism for read-only batches and switches to ordered sequential execution for batches containing mutating tools.
- `app/src/ai/agent/local/tool_card.rs`: add `apply_file_diff` call/result card mapping, preferably via existing `ApplyFileDiffs` proto types.
- `generate_openai_compatible_output`: continue appending the tool result back into `messages` and finalizing normally after successful or failed patch application.
