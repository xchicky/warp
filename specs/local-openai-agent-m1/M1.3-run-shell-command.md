# M1.3: Local OpenAI-Compatible Agent `run_shell_command`

## Summary

Replace suggestion-only shell handling with a feature-flagged `run_shell_command` capability that can execute commands after user approval, return bounded output to the local provider, and respect sandbox, timeout, and command-safety constraints.

Current branch note: PM referenced `crates/ai/src/agent/local/*`; this branch's local direct agent currently lives under `app/src/ai/agent/local/*`. If the module moves back under `crates/ai`, use the same entry points there.

## User Story

As a Warp user running a local OpenAI-compatible coding agent, I want the agent to run approved commands and consume their output, so that it can install-free verify changes, run targeted tests, inspect generated output, and finish tasks without manual command-copy handoff.

## Acceptance Criteria

1. `run_shell_command` is advertised only when `FeatureFlag::LocalAgentShellExecution` is enabled. The flag is added in `crates/warp_features/src/lib.rs`, re-exported through `crates/warp_core/src/features.rs`, and off by default.
2. `suggest_shell_command` remains available while M1.3 rolls out. When shell execution is disabled, the provider sees only `suggest_shell_command`; if it still calls `run_shell_command`, the client returns a disabled-by-flag tool error.
3. The tool schema accepts at minimum:
   - `command`: command string to execute.
   - `rationale`: optional explanation shown to the user.
   - `is_read_only`: optional model assertion.
   - `is_risky`: optional model assertion.
   - `cwd`: optional cwd override, resolved under the active writable root.
   - `timeout_ms`: optional timeout, clamped to a safe max.
4. Empty commands, commands containing NUL bytes, missing cwd for relative cwd resolution, cwd outside the approved root, and malformed arguments are rejected before user confirmation.
5. Execution requires visible user approval in M1.3. The confirmation surface shows command, cwd, rationale, read-only/risky flags, sandbox state, and timeout. Rejection returns a permission-denied/cancelled tool result to the provider.
6. The command is executed in the current terminal execution context when possible so Warp's existing shell, environment, denylist, and sandbox behavior remain the source of truth. Prefer reusing the existing `AIAgentActionType::RequestCommandOutput` / `ShellCommandExecutor` path rather than introducing an independent shell runner.
7. If the active terminal is already sandboxed, the command runs inside that sandbox. If it is not sandboxed, the confirmation clearly indicates unsandboxed local execution and does not autoexecute in M1.3.
8. Existing command denylist and permission checks still apply. User approval cannot bypass explicit denylist rules.
9. The command string is treated as an opaque shell command. The implementation must not compose it with unescaped model-provided fragments beyond existing Warp wrappers such as pager suppression.
10. Only one mutating or shell command batch runs at a time for the local provider. If the model emits multiple commands in one tool batch, execute them in provider order and require approval for each command.
11. Default timeout is conservative, and requested timeout is clamped to a max no higher than the existing `ShellCommandExecutor::MAX_AGENT_DELAY_DURATION` unless product explicitly approves otherwise.
12. On timeout, the command is stopped or control is transferred to the user using the existing terminal execution semantics. The provider receives a tool result that clearly states timeout/cancelled and includes bounded output captured so far.
13. Output returned to the provider is truncated by both byte and character limits. Truncation preserves the tail of output by default because test failures and final errors are usually at the end, and the result includes an explicit truncation marker.
14. Output shown in the terminal remains normal Warp terminal output; only the provider-facing tool result is truncated.
15. The result schema returned to the provider includes command, exit code when available, timed-out/cancelled/permission-denied state, and bounded stdout/stderr or combined terminal output.
16. Structured tool cards use the existing `RunShellCommand` call/result proto shape where practical so command history, YAML export, and existing UI affordances remain consistent.
17. `ActionResult` and `ResumeConversation` inputs continue routing to local conversations. If execution is implemented as an out-of-band action rather than inline tool execution, the local provider receives the command result on resume and then performs the finalize round.
18. Existing read-only tools, file-write tools, and the existing 44 unit tests continue to pass. Add unit tests for flag gating, schema validation, cwd validation, timeout clamping, truncation, user rejection result mapping, denylist result mapping, sequential command scheduling, and ActionResult-to-local-finalize conversion if used.

## Safety Risks

- Command injection: the model controls a shell command string. Mitigate by treating it as opaque, never concatenating trusted/untrusted fragments, and showing the exact command to the user before execution.
- Unapproved side effects: commands can modify or delete files, access network, or leak data. Mitigate with default-off flag, explicit confirmation, existing denylist, sandbox indication, and no M1.3 autoexecute for unsandboxed commands.
- Sandbox confusion: users may assume all local commands are sandboxed. Mitigate by surfacing sandbox state in confirmation and by running through the current terminal context when possible.
- Long-running commands: commands can hang or consume resources. Mitigate with timeout clamp, cancellation/control-transfer behavior, and one-at-a-time scheduling.
- Provider context blowup: command output can be huge. Mitigate with provider-facing truncation and explicit truncation markers.

## Scale Estimate

L. The tool schema is small, but safe execution crosses local-provider streaming, user approval, existing shell execution, ActionResult/resume behavior, and output shaping.

## PR Split Suggestion

Keep M1.3 as its own PR after file writes, with at most seven commits:

1. Add `LocalAgentShellExecution` feature flag, default off.
2. Add `run_shell_command` schema and validation with flag-off behavior.
3. Add local mutating/shell ordered scheduler coverage if not already complete.
4. Wire the tool to existing `RunShellCommand` / `RequestCommandOutput` confirmation and execution path.
5. Add ActionResult/resume-to-finalize handling for local shell results if the execution path is out-of-band.
6. Add provider-facing output truncation and timeout result shaping.
7. Add structured card/YAML consistency tests and update the local system prompt.

Do not mix M1.3 with M1.1/M1.2 implementation commits; it should be independently revertable.

## `ai::agent::local` Entry Points

- `app/src/ai/agent/local/mod.rs`: expose `run_shell_command` only when `LocalAgentShellExecution` is enabled.
- `app/src/ai/agent/local/mod.rs`: replace the current prompt language that says commands are only suggested when execution is enabled; keep suggestion-only language when the flag is off.
- `app/src/ai/agent/local/mod.rs`: add validation helpers for command, cwd, timeout clamp, and output truncation.
- `app/src/ai/agent/local/mod.rs`: route shell calls through an out-of-band action or an execution bridge rather than plain `spawn_blocking` direct shell execution, so user confirmation and sandbox state are preserved.
- `app/src/ai/agent/local/tool_card.rs`: add `run_shell_command` card/result mapping via existing `RunShellCommand` proto types.
- `app/src/ai/blocklist/action_model/execute/shell_command.rs`: reuse `ShellCommandExecutor` semantics for command execution, denylist, timeout, and long-running command handling where possible.
- `app/src/ai/agent/api/convert_conversation.rs` and local conversation resume code: ensure `ActionResult` for `RunShellCommand` is converted back into an OpenAI `tool` message and sent through the local finalize round.
