# M5.1: Local Agent Web Search and Fetch

## Summary

Add feature-flagged `web_search` and `web_fetch` tools for the local OpenAI-compatible agent. `web_search` uses a user-configured SearXNG endpoint. `web_fetch` performs bounded HTTP GET requests and converts supported content into provider-visible text snippets with citations/URLs.

Current branch note: Warp already has `WebSearchStatus`, `WebFetchStatus`, `WebSearchView`, `WebFetchView`, execution-profile web search permissions, and `FeatureFlag::WebSearchUI` / `FeatureFlag::WebFetchUI`. The local OpenAI-compatible agent does not currently execute local web search/fetch tools.

## User Story

As a Warp user using a local OpenAI-compatible agent, I want the agent to search the web and fetch pages when I explicitly enable the capability, so it can answer current-information questions without switching to a remote Warp agent or leaking my queries into product telemetry.

## Acceptance Criteria

1. Add `FeatureFlag::LocalAgentWeb`, default off, following the existing local-agent feature flag pattern.
2. When `LocalAgentWeb` is off, local OpenAI-compatible tool definitions and request behavior are unchanged.
3. When the flag is on, advertise two OpenAI-compatible function tools:
   - `web_search`
   - `web_fetch`
4. `web_search` accepts bounded arguments:
   - `query`: required non-empty string.
   - Optional `max_results`, clamped to a small maximum.
   - Optional `time_range` only if it maps cleanly to SearXNG parameters; otherwise omit from V0.
5. `web_fetch` accepts bounded arguments:
   - `url`: required absolute HTTP(S) URL.
   - Optional `max_bytes` or `max_chars`, clamped to the implementation maximum.
6. `web_search` uses a user-configured SearXNG endpoint. If no endpoint is configured, return a provider-visible unavailable error with setup guidance. Do not silently call Brave, DuckDuckGo, Google, Bing, or Warp services by default.
7. SearXNG configuration must be local-only. Prefer a local settings or preferences location that is not cloud-synced; if an existing settings system is reused, the PR description must state whether the endpoint syncs and why that is acceptable.
8. `web_search` requests use SearXNG JSON output and return bounded result metadata:
   - title
   - URL
   - snippet/content summary
   - source engine if provided
   - published date if provided
9. `web_fetch` performs HTTP GET only. It must not execute JavaScript, submit forms, reuse browser cookies, authenticate, or crawl linked pages.
10. `web_fetch` allows HTTPS by default. Plain HTTP is allowed only with explicit result metadata noting the insecure scheme and after all SSRF checks pass; if implementation chooses to deny HTTP in V0, the provider-visible error must be clear.
11. `web_fetch` must deny SSRF-sensitive targets before sending the request and after redirects:
   - loopback and localhost names
   - RFC1918 private IPv4 ranges
   - link-local IPv4 and IPv6 ranges
   - unique-local IPv6 ranges
   - metadata services such as `169.254.169.254`
   - `.local` and other local network hostnames where they can be identified
12. Resolve hostnames and apply IP deny checks to all resolved addresses. Redirects must be bounded and rechecked at every hop.
13. Enforce request timeout, redirect limit, response byte limit, and text character limit. Partial results must be marked as truncated.
14. Content types are allowlisted:
   - `text/html`
   - `text/plain`
   - `application/json`
   - `application/xml` / `text/xml` if conversion is straightforward
   Unsupported binary/media content returns bounded metadata, not raw bytes.
15. HTML conversion strips scripts, styles, hidden boilerplate where practical, and returns readable text. Do not add a heavy browser or readability dependency unless review proves it is necessary.
16. Tool results returned to the provider include source URLs and bounded content. Use byte and character limits similar to MCP result hardening.
17. Emit existing `AIAgentOutputMessageType::WebSearch` / `WebFetch` statuses so existing UI cards can render searching/fetching/success/error where practical.
18. Web queries, URLs, page content, snippets, and fetched text must not enter product telemetry, Warp billing, AIRequestUsageModel, unsafe logs, or crash breadcrumbs. Logs may include coarse status and redacted error classes only.
19. `robots.txt` crawling is a non-goal for M5.1 because `web_fetch` is a single requested URL fetch, not an automated crawler. The PR description should explicitly state this boundary.
20. Plan mode handling:
   - `web_search` may be allowed in plan mode because it reads public data.
   - `web_fetch` may be allowed in plan mode for HTTP(S) reads.
   - Both still require `LocalAgentWeb` and must not bypass plan-mode forbidden mutating tools.
21. Mixed tool batches preserve provider order. Web tools are read-only and may run in parallel with other read-only tools, but must not reorder around existing sequential/mutating tools.
22. Tool-result resume/finalize rounds remain routed to the local provider and preserve the same local conversation id.
23. Existing local-agent tests continue to pass. Add focused tests for:
   - Feature flag off omits tools.
   - Feature flag on advertises schemas.
   - Missing SearXNG endpoint returns unavailable error.
   - SearXNG JSON is converted into bounded search results.
   - Fetch denies loopback, RFC1918, link-local, `.local`, and metadata hosts.
   - Fetch rechecks redirect targets.
   - Fetch truncates by byte and character limits.
   - Binary content returns metadata only.
   - Web output is converted into existing UI message/status types.

## Safety Risks

- SSRF: provider or prompt injection could ask for private network URLs. Mitigate with URL parsing, DNS resolution checks, redirect rechecks, and fail-closed deny lists.
- Privacy leakage: search queries and fetched URLs can reveal user intent. Mitigate with local SearXNG configuration and no product telemetry.
- Context blowup: pages can be large. Mitigate with byte/char limits, content-type allowlist, and truncation metadata.
- Malware/untrusted content: fetched pages may contain adversarial prompt text. Mitigate by labeling fetched content as untrusted source material in provider context.
- Dependency creep: browser automation would greatly expand scope. Mitigate by keeping M5.1 to simple HTTP GET and HTML/text conversion.

## Scale Estimate

M/L. Existing UI status types exist, but local web execution adds new outbound HTTP configuration, SSRF hardening, parsing, output truncation, and tests.

## PR Split Suggestion

Keep M5.1 as one implementation PR if the diff remains contained; otherwise split SearXNG search and HTTP fetch into separate PRs. Target at most seven commits:

1. Add `LocalAgentWeb` feature flag, default off.
2. Add local settings/config lookup for SearXNG endpoint.
3. Add `web_search` and `web_fetch` OpenAI-compatible schemas.
4. Implement SearXNG JSON search and bounded result conversion.
5. Implement HTTP fetch safety, content-type handling, HTML/text conversion, and truncation.
6. Reuse existing web status messages/UI conversion and local resume handling.
7. Add tests for schemas, SSRF denial, redirects, truncation, search conversion, and logging/privacy boundaries.

Each commit should compile independently. Do not add a new browser dependency in this PR.

## Entry Points

- `app/src/ai/agent/local/mod.rs`: local tool definitions, argument parsing, execution, result conversion, scheduling, and local tests.
- `app/src/ai/agent/mod.rs`: `WebSearchStatus`, `WebFetchStatus`, and output message display.
- `app/src/ai/agent/api/convert_from.rs`, `convert_to.rs`, and `convert_conversation.rs`: history/status conversion if local web messages need serialization.
- `app/src/ai/blocklist/inline_action/web_search.rs` and `web_fetch.rs`: existing UI cards.
- `app/src/ai/blocklist/block.rs` and `view_impl/output.rs`: existing web status view wiring.
- `app/src/ai/execution_profiles/*` and `app/src/ai/blocklist/permissions.rs`: existing web-search enablement concepts, if reused for local config gating.
- `crates/http_client/src/lib.rs` or existing `reqwest` client usage: HTTP execution wrapper.
- `app/Cargo.toml`, `app/src/lib.rs`, `crates/warp_features/src/lib.rs`: feature flag and dependency plumbing.
