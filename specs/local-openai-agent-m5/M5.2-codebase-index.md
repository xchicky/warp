# M5.2: Local Agent Codebase Index and Search

## Summary

Add a feature-flagged local codebase search/index capability for the local OpenAI-compatible agent. The V0 should stay local by using a lexical/symbol/ripgrep-backed index or adapter, reusing existing SearchCodebase action/result/UI surfaces where practical. It must not upload source code or rely on Warp cloud embeddings/vector storage.

Current branch note: Warp already has `AIAgentActionType::SearchCodebase`, `SearchCodebaseRequest`, `SearchCodebaseResult`, `SearchCodebaseExecutor`, `GetRelevantFilesController`, `SearchCodebaseView`, and `CodebaseIndexManager` under `crates/ai/src/index/full_source_code_embedding`. The existing full-source embedding flow is cloud-store oriented and is not automatically acceptable for local-agent M5 unless a local-only store/client path is provided.

## User Story

As a Warp user using a local OpenAI-compatible agent in a repository, I want the agent to search and reason over indexed code structure, so it can answer questions like "where is this function called?" or "find similar implementations" without rereading the whole project every round.

## Acceptance Criteria

1. Add `FeatureFlag::LocalAgentCodebaseIndex`, default off, following the existing local-agent feature flag pattern.
2. When `LocalAgentCodebaseIndex` is off, local OpenAI-compatible tool definitions and request behavior are unchanged.
3. When the flag is on, advertise one or two OpenAI-compatible function tools:
   - Required: `search_codebase`
   - Optional: `find_references` if references can be implemented reliably without a language server dependency.
4. `search_codebase` accepts bounded arguments:
   - `query`: required non-empty string.
   - Optional `path` / `scope`, constrained to the current workspace or project root.
   - Optional `max_results`, clamped to a small maximum.
   - Optional `include_snippets`, default true with bounded snippet size.
5. `find_references`, if included, accepts:
   - `symbol`: required non-empty string.
   - Optional `path` / `scope`, constrained to the current workspace or project root.
   - Optional `max_results`, clamped.
6. Tool names must not conflict with existing local tools or MCP namespaced tools. If reusing the existing `SearchCodebase` action name/path, document the local-agent mapping in the PR.
7. Source content and index artifacts stay local:
   - Do not upload source files, fragments, embeddings, or index metadata to Warp services.
   - Do not use the existing cloud embedding store/vector DB for local-agent M5.
   - Do not emit index contents into product telemetry, Warp billing, unsafe logs, or crash breadcrumbs.
8. The V0 index may be lexical/symbol-based instead of semantic embeddings. It should use one of:
   - existing ripgrep/global search primitives,
   - a local persisted token/symbol index,
   - or a local-only adapter around existing index abstractions.
9. Persistence must be local-only and outside the repository by default. Do not write index files into git worktrees. Prefer existing app cache/data directories; if SQLite/Diesel is used, document schema ownership and cleanup policy.
10. Respect project/workspace boundaries:
   - Canonicalize requested paths.
   - Deny paths outside the active project/workspace root.
   - Respect symlink escapes.
   - Respect ignore rules where existing search/index infrastructure already does.
11. Indexing must enforce file count, file size, directory depth, and total byte limits. Large/binary/generated files should be skipped with bounded status metadata.
12. Incremental updates are required for files changed through Warp-local mutating tools where practical. At minimum, stale index state must be detected and surfaced in tool results with guidance to rebuild/refresh.
13. First-run indexing must be user-visible and cancellable if it can take noticeable time. Do not block the UI indefinitely.
14. Tool results returned to the provider are bounded:
   - path
   - line range or symbol location if available
   - small snippet
   - match/relevance score if available
   - stale/truncated status when applicable
15. Preserve provider order in mixed tool batches. Codebase search is read-only and may run in parallel with other read-only tools, but must not reorder around sequential/mutating tools.
16. Plan mode may advertise `search_codebase` and `find_references` because they are read-only. They must still obey path containment and size limits.
17. If no project root/current working directory is available, return a provider-visible error. Do not search the user's home directory by default.
18. If the index is unavailable, stale, or still building, return bounded provider-visible status. Do not silently fall back to uploading source or remote codebase search.
19. Reuse existing `SearchCodebaseResult` / `SearchCodebaseView` where practical so UI behavior remains familiar. If a separate local result type is necessary, keep conversion to existing display primitives narrow.
20. Existing local-agent tests continue to pass. Add focused tests for:
   - Feature flag off omits tools.
   - Feature flag on advertises schemas.
   - Path scope canonicalization rejects `..` and symlink escapes.
   - Large/binary files are skipped or bounded.
   - Search results are bounded and deterministic.
   - Stale index state is surfaced.
   - No cloud store/client is called in local-agent index path.
   - Plan mode allows codebase search while still denying mutating tools.

## Safety Risks

- Source exfiltration: existing embedding paths can send code to remote services. Mitigate by forbidding cloud store/vector DB use for M5 local-agent indexing.
- Hidden persistence: index artifacts could land in the repository or telemetry. Mitigate by using app cache/data directories and no product telemetry.
- Path escape: provider-supplied paths could escape the workspace. Mitigate with canonicalization, root containment, and symlink checks.
- Resource exhaustion: large repositories can consume CPU, memory, or disk. Mitigate with limits, cancellation, incremental indexing, and bounded results.
- Stale context: agent decisions based on stale index data can be wrong. Mitigate with staleness detection and visible status.

## Scale Estimate

L. Existing search UI/action primitives exist, but a local-stay-local index path crosses storage, file watching or stale detection, path safety, result conversion, and performance limits.

## PR Split Suggestion

Prefer two implementation PRs if the diff is large:

1. Foundation PR:
   - Add `LocalAgentCodebaseIndex` feature flag.
   - Add local index/search adapter interface.
   - Add local persistence/cache path and path safety helpers.
   - Add index build/refresh/stale status tests.
2. Tool integration PR:
   - Add `search_codebase` schema and optional `find_references`.
   - Wire local tool execution and bounded result conversion.
   - Reuse existing `SearchCodebase` UI/action/history where practical.
   - Add local-agent provider resume and plan-mode tests.

If kept as one PR, cap it at eight commits and keep each commit independently testable.

## Entry Points

- `app/src/ai/agent/local/mod.rs`: local tool definitions, execution, result summarization, plan-mode tool policy, and local tests.
- `crates/ai/src/index/full_source_code_embedding/*`: existing index/chunking/manager code to evaluate for reusable local-only pieces.
- `app/src/ai/get_relevant_files/controller.rs`: existing retrieval controller and event model.
- `app/src/ai/blocklist/action_model/execute/search_codebase.rs`: existing `SearchCodebaseExecutor` and action result path.
- `crates/ai/src/agent/action/mod.rs` and `action_result/mod.rs`: `SearchCodebaseRequest` / `SearchCodebaseResult` types.
- `app/src/ai/blocklist/inline_action/search_codebase.rs`: existing SearchCodebase UI.
- `app/src/workspace/view/global_search/model.rs` and `warp_ripgrep::search`: existing lexical search primitives.
- `app/src/ai/agent/api/convert_from.rs`, `convert_to.rs`, and `convert_conversation.rs`: history conversion if local codebase search reuses action results.
- `app/Cargo.toml`, `app/src/lib.rs`, `crates/warp_features/src/lib.rs`: feature flag and dependency plumbing.
