# M5.3: Local Agent Subagent Delegation

## Summary

Add a feature-flagged local `spawn_subagent` capability so the local OpenAI-compatible agent can delegate a bounded task to an isolated child agent context and receive a summarized result. The child uses the same local provider route/config, feature flags, plan-mode restrictions, and user approval gates as the parent. It must not start an OS process or bypass existing tool permissions.

Current branch note: Warp already has orchestration primitives such as `AIAgentActionType::StartAgent`, `RunAgents`, `SendMessageToAgent`, `StartAgentExecutor`, `RunAgentsExecutor`, `CLISubagentController`, child agent views, and subagent output types. M5.3 should reuse these surfaces where practical, but local OpenAI-compatible child execution should be a single-process task/actor model.

## User Story

As a Warp user using a local OpenAI-compatible agent, I want the agent to delegate isolated research or implementation subtasks to child contexts, so complex work can be decomposed without polluting the main conversation history or granting hidden extra permissions.

## Acceptance Criteria

1. Add `FeatureFlag::LocalAgentSubagent`, default off, following the existing local-agent feature flag pattern.
2. When `LocalAgentSubagent` is off, local OpenAI-compatible tool definitions and request behavior are unchanged.
3. When the flag is on, advertise one OpenAI-compatible function tool named `spawn_subagent`.
4. `spawn_subagent` accepts bounded arguments:
   - `task`: required non-empty string describing the child task.
   - Optional `context`: bounded string with explicit parent-selected context.
   - Optional `tools`: bounded list of requested tool names from the parent's currently allowed tool set.
   - Optional `label`: bounded display label.
5. V0 should run synchronously from the parent provider's perspective: the parent receives one bounded result or error before the local provider finalize round continues. If implementation chooses async, it must preserve the same safety guarantees and explain the decision in the PR.
6. A child agent has an isolated conversation id/history. It must not inherit the full parent conversation transcript by default. Only `task`, bounded `context`, and explicitly selected safe summaries may be passed in.
7. The child uses the same local OpenAI-compatible provider configuration as the parent unless the user explicitly chooses otherwise through existing settings. It must not fall back to remote/Oz.
8. The child inherits the same feature flags and permission policy as the parent:
   - It cannot enable `LocalAgentFileWrites`, `LocalAgentShellExecution`, `LocalAgentMcp`, `LocalAgentWeb`, `LocalAgentCodebaseIndex`, or other local flags that are off for the parent.
   - It cannot gain tools that the parent did not have available.
9. Plan mode restrictions apply to subagents:
   - If the parent is in plan mode, `spawn_subagent` may only create a read-only planning/research child.
   - The child must deny mutating tools, MCP tools, shell execution, and file writes just as the parent would.
   - A child cannot call `exit_plan_mode` on behalf of the parent.
10. All child tool calls that require visible user approval still use the existing main-process approval gate. Subagents must not auto-approve shell commands, file writes, MCP calls, or other risky actions.
11. If `tools` is supplied, validate every requested tool against the currently advertised/allowed parent tool catalog. Unknown, hidden, forbidden, or MCP namespaced tools denied by current policy fail closed.
12. Prevent recursive runaway:
   - Cap subagent depth, with V0 recommended depth 1.
   - Cap number of child agents per parent round.
   - Cap child provider rounds and wall-clock timeout.
13. Child results returned to the parent are bounded:
   - summary
   - relevant findings
   - files/URLs/tools touched as metadata only where applicable
   - errors/cancellations
   Do not inject full child transcripts into the parent provider context.
14. Child conversation/tool history may be visible in UI using existing subagent surfaces where practical, but must be clearly labeled as child output.
15. Child todo behavior must be explicit:
   - A child may read the parent's current todo snapshot as bounded context if useful.
   - A child must not mutate the parent's todo list unless it calls the same `todo_write` tool through the normal approval/policy path and the parent policy allows it.
16. Mixed tool batches preserve provider order. Treat `spawn_subagent` as sequential/control-flow work: it must not run in parallel with mutating tools, MCP calls, shell execution, or another subagent spawn unless M5.3 explicitly adds parallel child orchestration with limits.
17. If the user cancels the parent run, active child work must be cancelled or marked cancelled and reported with bounded status.
18. Tool-result resume/finalize rounds remain routed to the local provider and preserve the parent local conversation id.
19. Product telemetry must not include task text, child context, child transcript, tool arguments, file contents, web URLs, or provider output. Coarse local-only logs may include redacted status and duration.
20. Existing local-agent tests continue to pass. Add focused tests for:
   - Feature flag off omits `spawn_subagent`.
   - Feature flag on advertises the expected schema.
   - Child context excludes full parent history by default.
   - Child inherits feature flags and denied tools.
   - Plan-mode child denies mutating tools and cannot exit parent plan mode.
   - Unknown/forbidden `tools` arguments fail closed.
   - Depth/round/time limits stop recursive runaway.
   - Child result is bounded before being appended to parent context.
   - Parent resume stays local with the same conversation id.

## Safety Risks

- Permission bypass: a child could gain tools or flags not available to the parent. Mitigate with inherited policy and tool catalog validation.
- Context leakage: passing full parent history into children can leak sensitive context and bloat tokens. Mitigate with explicit bounded task/context only.
- Runaway recursion: subagents spawning subagents can exhaust resources. Mitigate with depth, count, round, and timeout caps.
- Approval confusion: users may not realize a child triggered a risky action. Mitigate by routing all approvals through existing visible main-process gates and labeling child actions.
- UI ambiguity: child output could look like parent output. Mitigate with existing subagent cards or clear labels.

## Scale Estimate

L. Existing orchestration primitives exist, but local OpenAI-compatible child execution crosses provider routing, tool policy inheritance, cancellation, bounded context/result conversion, UI/status reuse, and tests.

## PR Split Suggestion

Prefer two implementation PRs unless the diff stays small:

1. Local subagent runtime PR:
   - Add `LocalAgentSubagent` feature flag.
   - Add isolated child context model and inherited tool policy.
   - Add depth/round/timeout/cancellation limits.
   - Add tests for context isolation and policy inheritance.
2. Tool/UI integration PR:
   - Add `spawn_subagent` schema and local execution path.
   - Reuse existing subagent action/result/UI surfaces where practical.
   - Add bounded child result conversion and local resume/finalize handling.
   - Add tests for forbidden tools, plan mode, cancellation, and ordered scheduling.

If kept as one PR, cap it at eight commits and keep each commit independently testable.

## Entry Points

- `app/src/ai/agent/local/mod.rs`: local tool definition, child provider invocation, policy inheritance, scheduling, resume/finalize, and local tests.
- `crates/ai/src/agent/action/mod.rs` and `action_result/mod.rs`: existing `StartAgent`, `RunAgents`, `SendMessageToAgent`, and related result types.
- `app/src/ai/blocklist/action_model/execute/start_agent.rs` and `run_agents.rs`: existing out-of-band orchestration executors to evaluate for reuse.
- `app/src/ai/blocklist/block/cli_controller.rs`: `CLISubagentController` and child state events.
- `app/src/ai/blocklist/inline_action/run_agents_card_view.rs` and related orchestration UI: existing subagent approval/status display.
- `app/src/pane_group/pane/terminal_pane.rs`: existing StartAgent local/remote conversation creation paths, if child UI reuse requires it.
- `app/src/ai/agent/api/convert_from.rs`, `convert_to.rs`, and `convert_conversation.rs`: history conversion for child tool calls/results.
- `app/src/ai/agent/local/skills.rs`: useful pattern for bounded prompt-as-data context if child context injection needs a helper.
- `app/Cargo.toml`, `app/src/lib.rs`, `crates/warp_features/src/lib.rs`: feature flag plumbing.
