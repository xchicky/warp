# M3.1: Local Agent Image Input Passthrough

## Summary

Allow the local OpenAI-compatible agent to include user-attached images in provider requests when `LocalAgentImageInput` is enabled and the selected local model is considered vision-capable. Reuse Warp's existing image attachment path and encode images as OpenAI-compatible `image_url` data URLs in user messages.

Current branch note: images already exist as `AIAgentContext::Image(ImageContext)`, but `app/src/ai/agent/local/mod.rs` drops them and uses text-only `OpenAIChatMessage.content`.

## User Story

As a Warp user using a local vision-capable OpenAI-compatible provider, I want images that I paste or drag into Warp to be sent with my local agent request, so I can ask the local model to inspect screenshots, UI states, diagrams, and visual errors without switching to Warp-hosted models.

## Acceptance Criteria

1. Add `FeatureFlag::LocalAgentImageInput`, default off, following the M1/M2 feature flag pattern:
   - `crates/warp_features/src/lib.rs`
   - `crates/warp_core/src/features.rs` if needed by the local feature bridge
   - `app/Cargo.toml`
   - `app/src/lib.rs`
2. When `LocalAgentImageInput` is off, local OpenAI-compatible requests continue to ignore `AIAgentContext::Image(_)` exactly as they do after M2.4. Existing text-only request JSON remains unchanged.
3. When the flag is on, local direct extracts images from the current user query context:
   - Images attached through paste/drag/drop and stored as `AIAgentContext::Image`.
   - No direct filesystem scanning or new attachment source in M3.1.
   - No image extraction from MCP tool results or local tool results.
4. Supported MIME types are `image/png`, `image/jpeg`, `image/jpg`, `image/webp`, and `image/gif`, matching `SUPPORTED_IMAGE_MIME_TYPES`. Unsupported image MIME types produce a provider-visible, user-friendly local error instead of silently dropping the image.
5. Enforce existing image count and size limits from `app/src/util/image.rs`:
   - Maximum image count per user query uses `MAX_IMAGE_COUNT_FOR_QUERY`.
   - Per-image byte limits use `MAX_IMAGE_SIZE_BYTES` after processing.
   - Invalid base64, missing MIME type, oversize, or decode failures fail before the provider request.
6. Normalize accepted images before sending:
   - Reuse and extend `process_image_for_agent` or an adjacent helper.
   - Strip EXIF and other metadata that may contain location or device information before constructing the data URL.
   - If a format cannot be safely normalized, fail closed with a clear error.
7. Convert `OpenAIChatMessage.content` from text-only to a tagged representation that can serialize either:
   - A string for text-only messages.
   - An array of content parts for multimodal user messages.
8. Multimodal user messages preserve the text currently generated by `openai_messages_from_inputs_and_tasks` as the first content part, then append one `image_url` part per image:
   - `{"type":"text","text":"..."}`
   - `{"type":"image_url","image_url":{"url":"data:<mime>;base64,<data>"}}`
9. System, assistant, and tool messages stay text-only unless a future milestone explicitly expands them.
10. Provider capability gating must run before sending image bytes:
   - Prefer existing model metadata (`LLMPreferences::vision_supported`) when available.
   - For local custom OpenAI-compatible models where Warp marks `vision_supported` true by default, rely on the feature flag plus provider error handling; do not introduce a network probe in M3.1.
   - If the selected model is known not to support vision, return a clear local error and do not call the provider.
11. If the provider rejects multimodal payloads with a clear unsupported-image or unsupported-content error, surface that error to the user and finish the local request without retrying without images. Silent fallback would answer the wrong question.
12. Data URLs, raw base64 image bytes, filenames, and image-derived metadata must not be written to logs, debug output, telemetry, or tool result summaries.
13. History conversion must not duplicate images on resume/finalize rounds:
   - User images are included only for the user query that supplied them.
   - Tool-result finalize rounds after shell/MCP/local tools do not resend prior image bytes unless the current input still contains them by design.
14. Existing local tools, MCP passthrough, shell execution, file writes, and M2 tests continue to pass.
15. Add focused tests for:
   - Flag off keeps text-only request shape.
   - Flag on serializes text plus image parts.
   - Multiple images preserve order.
   - Unsupported MIME and oversize images fail before provider request.
   - Images are not included in tool messages or logs/test debug output.
   - Known non-vision model path fails with a clear error.

## Safety Risks

- Image privacy leak: screenshots may include secrets or personal data. Mitigate with explicit user attachment source, default-off flag, and no logs/telemetry.
- EXIF/location leakage: original JPEG/WebP metadata may include GPS or device information. Mitigate by normalizing and stripping metadata before provider request.
- Provider mismatch: non-vision providers may reject or ignore images. Mitigate with capability gating and clear no-retry errors.
- Context blowup: base64 inflates request size. Mitigate with image count, byte, and processing limits before constructing request JSON.
- Accidental replay: images could be resent on tool finalize/resume rounds. Mitigate with tests around message history and current input handling.

## Scale Estimate

M. Existing image attachment and processing infrastructure exists, but local direct needs a request schema change from string-only chat messages to multimodal content parts plus careful validation and tests.

## PR Split Suggestion

Keep M3.1 as one PR with at most six commits:

1. Add `LocalAgentImageInput` feature flag, default off.
2. Add local image collection and validation helpers around `AIAgentContext::Image`.
3. Add metadata-stripping image normalization or extend existing image processing helpers.
4. Change local OpenAI message serialization to support string or content-part arrays.
5. Add capability and provider-error handling for unsupported vision payloads.
6. Add unit tests for request shape, limits, flag gating, and no image leakage.

Each commit should compile independently. Avoid touching shared remote/Oz image behavior except for reusable image helper tests.

## Entry Points

- `app/src/ai/agent/local/mod.rs`: `OpenAIChatMessage`, `openai_messages_from_inputs_and_tasks`, `local_context_section`, request construction, and local tests.
- `app/src/ai/blocklist/context_model.rs`: existing pending image to `AIAgentContext::Image` path; should not need behavioral changes unless tests require fixtures.
- `app/src/util/image.rs`: image MIME whitelist, count/size constants, processing, and metadata stripping.
- `app/src/ai/llms.rs`: model `vision_supported` metadata and local custom model behavior.
- `app/src/ai/agent/api.rs`: thread any needed vision-capability bit into `LocalDirectConfig` or local request params.
- `app/Cargo.toml`, `app/src/lib.rs`, `crates/warp_features/src/lib.rs`: feature flag plumbing.
